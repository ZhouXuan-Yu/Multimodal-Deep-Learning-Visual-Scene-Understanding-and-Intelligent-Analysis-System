# 五、框架设计

**本章承接第三章系统总体设计与第四章核心算法设计，从分布式计算、视觉感知、数据可视化与系统容错四个技术方向进行文献脉络梳理与框架设计论证，为整体技术方案提供理论层面的合法性支撑。**

## 5.1 端云协同架构研究

端云协同架构的理论基础包含两条主线：一是基于 Flynn 分类法的分布式计算范式，Kubeflow、TensorFlow Serving 等"模型即服务"方案为云端推理提供了统一部署与弹性扩展的工程参考 [cite: 1, 3, 4]；二是微内核架构的"核心最小化、服务外置化"原则，指导本研究将 YOLO 推理、ResNet 属性识别等计算密集型任务封装为独立服务模块，实现低耦合独立部署 [cite: 2]。

围绕模型部署模式，学术界存在集中式推理与分布式推理的持续争议。集中式方案通过云端 GPU 集群统一管理模型，但面临网络带宽瓶颈与延迟敏感性；分布式方案将推理前移至边缘，能够缓解数据传输压力，但边缘设备的算力限制迫使模型在精度与速度之间做出更严苛的权衡 [cite: 5, 6, 7]。模型量化研究进一步表明，INT8 量化在多数视觉任务上可将推理速度提升 $2$–$3$ 倍，但在年龄估计等细粒度属性分类任务上，量化对类别边界的影响更为显著 [cite: 8]。

现有研究的空白在于：如何在不同算力层级之间实现异构任务的统一调度。本研究中路径规划（秒级延迟）、人物识别（百毫秒级延迟）、监控数据（秒级轮询）与智能咨询（秒级 API 调用）四类任务在响应时序与容错要求上存在显著差异，这一多时序多约束的异构调度问题在现有文献中缺乏针对性研究。面向未来，端云感知卸载与推理计划驱动的自适应协同被认为是下一代智能系统的核心技术方向 [cite: 10, 11]。

## 5.2 多模型视觉感知流水线研究

多模型视觉感知流水线的核心理论之一是误差传播机制。Garcia 等人于 1997 年的研究表明，在三阶段流水线中若各阶段准确率分别为 95%、92% 与 90%，端到端正确率将降至约 79%，且提升准确率最低的瓶颈阶段对端到端性能的边际收益远高于优化高水平阶段 [cite: 12]。

目标检测领域形成了从两阶段范式到单阶段范式的清晰演进。Faster R-CNN 等两阶段方法在精度上具有优势但推理速度受限 [cite: 14]；YOLO 将检测重新表述为单一回归问题，在精度与速度之间取得了良好平衡 [cite: 13]。YOLOv8 通过 CSP 骨干网络与 PAN 颈部结构，在 COCO 数据集上达到 $AP_{50}$ 为 80.3% 的精度，同时在消费级 GPU 上保持约 150 FPS 的推理速度，本研究选择 YOLO 作为人物检测核心引擎正是基于此权衡考量。ResNet 通过残差连接解决了深层网络梯度消失问题，ResNet50 较深的层次适合提取年龄相关的宏观纹理特征，ResNet18 较浅的层次适合捕捉服装颜色的局部色彩分布 [cite: 15]。

当前争议集中于端到端统一建模与模块化流水线的取舍：DETR 等统一建模方案无需人工设计的锚框与 NMS 后处理，但收敛速度慢且小数据集下性能不如 YOLO [cite: 16, 17]；模块化流水线允许各模块独立训练、独立优化与独立部署，在本科毕业设计的迭代开发场景下具有显著维护优势。研究空白在于置信度感知的误差主动控制机制——当某属性识别置信度低于阈值时，应选择"返回未知标签"还是"返回概率最高标签"，现有文献缺乏将置信度决策与用户满意度关联建模的量化研究。本研究通过显式置信度字段在流水线中的传递，为该方向提供了可检验的实验基础。

## 5.3 数据可视化与地理信息服务研究

地理信息科学理论为智航监控模块的空间数据管理提供了概念框架。Goodchild 等学者提出的"地理信息科学"框架将空间数据与属性数据的统一管理作为核心特征 [cite: 21]；W3C 的 WMTS 标准与 OGC 的 WMS 标准为互联网地图服务提供了互操作协议基础 [cite: 22, 23]。

多维数据可视化领域的经典理论为图表设计提供了认知科学层面的指导。Tufte 于 1983 年提出的"数据墨水比"原则主张最大化承载数据信息的视觉元素比例 [cite: 24]；Shneiderman 提出的"视觉信息搜索 mantra"（overview first, zoom and filter; then details-on-demand）将用户信息搜索行为划分为概览、缩放过滤与按需获取详情三个阶段，对监控模块"总览图表→筛选条件→详情弹窗"的交互流程设计具有直接指导意义 [cite: 25]。

现有争议聚焦于空间维度与时序维度的表达优先级。传统 SCADA 系统以时序数据为核心，地图仅作辅助参考 [cite: 26]；随着无人机等移动监控设备的普及，"空间态势感知"的价值日益凸显，推动监控可视化从"时序主控"向"时空并重"转变 [cite: 27]。研究空白在于缺乏面向视频流、时序指标、空间轨迹与告警事件等多模态异构数据的统一可视化框架。面向未来，数字孪生与增强现实驱动的沉浸式空间数据可视化被认为是下一代监控平台的技术方向 [cite: 31, 32]。

## 5.4 智能咨询与容错机制研究

智能咨询模块的容错设计建立在软件可靠性工程的经典理论基础之上。Lebresco 可靠性度量框架中的失效率 $\lambda(t)$、均值无故障时间 MTBF 与可用度 $A = \frac{\text{MTBF}}{\text{MTBF} + \text{MTTR}}$ 等概念为评估端到端链路可靠性提供了量化工具 [cite: 34]。CAP 定理的形式化分析表明，在网络分区不可避免的现实环境中，系统必须在一致性与可用性之间做出权衡，接受最终一致性比追求强一致性更符合工程实际 [cite: 38, 39]。

容错架构领域，Avizienis 等学者提出的"多样化冗余"原则主张在关键环节部署功能等效但实现方式不同的冗余组件以降低共因失效风险 [cite: 35]。Paxos、Raft 等共识协议中"多数派确认"的降级策略——当无法获得多数派响应时系统进入"只读模式"而非完全不可用——与本研究"DeepSeek 失败则降级至模板"的设计思路具有内在一致性：两者都将"优雅降级"作为可靠性设计的核心目标 [cite: 36]。

当前争议集中于强一致性保证与优雅降级两种设计哲学的对立。强一致性方案以 DynamoDB、Spanner 为代表，主张通过同步复制确保数据严格一致；本研究采用的"三层递进容错"策略体现了"优雅降级"哲学：DeepSeek API 失败时降级至本地模板，模板解析失败时降级至行级扫描，邮件发送失败时保留摘要内容供用户手动获取。研究空白在于大模型输出的不确定性尚未在学术文献中得到充分解答：如何从系统架构层面设计"解析失败后的自动恢复"机制？未来趋势指向"自主错误恢复"（如 Constitutional AI、Reflexion 框架）与"不确定性感知系统"（如概率编程框架）[cite: 40, 41, 42]。

## 5.5 综述总结与框架设计决策

综合四个技术方向的文献脉络梳理，本研究的框架设计决策如下。

在端云协同架构方面，本设计采用"前端富交互+轻量后端服务+本地模型+外部 API"的混合式分层架构，以 FastAPI 作为统一调度层，以 Ollama 本地大模型处理语义解析，以 YOLO/ResNet 本地模型处理视觉推理，以云端 DeepSeek API 处理智能咨询，通过三层降级策略保障系统在部分组件失效时的基本可用性。

在视觉感知流水线方面，本设计采用"检测—属性识别—置信度融合"三阶段模块化流水线，以 YOLOv8 为人物检测核心，以 ResNet50/ResNet18 为属性识别骨干网络，通过置信度阈值机制实现属性省略，以统一 JSON 接口封装多模型输出。各模块可独立训练、独立优化与独立部署，兼顾系统级鲁棒性与工程可维护性。

在数据可视化方面，本设计采用"地图—图表—侧边栏"三区联动布局，以 ECharts 为统一图表渲染引擎，以高德地图 JS API 为空间数据底座，通过 GeoApiDashboard 组件实现空间可见集与数据过滤条件的双向联动，在有限的屏幕空间内组织空间、时序、事件等多种模态的信息呈现。

在容错机制方面，本设计采用"DeepSeek API → 本地模板 → 固定回退"三层递进容错链路。设三阶段成功概率分别为 $p_1, p_2, p_3$，端到端完全成功的概率为 $P_{\text{full success}} = p_1 \cdot p_2 \cdot p_3$；加入容错链路后可用性下界可扩展为 $P_{\text{end-to-end}} \geq 1 - (1 - p_1 \cdot p_2)(1 - p_1' \cdot p_2) \cdot (1 - p_3)$。当 $p_1 = 0.9$、$p_1' = 0.95$、$p_2 = 0.85$、$p_3 = 0.98$ 时，端到端成功率可从 75% 提升至 90% 以上，验证了多样化冗余策略的有效性。

上述四方面框架设计决策相互支撑，共同构成本设计"分层解耦、迭代闭环"总体架构的技术基石。
