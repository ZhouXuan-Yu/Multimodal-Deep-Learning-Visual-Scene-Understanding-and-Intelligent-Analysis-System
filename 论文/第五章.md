五、系统实现

5.1 开发环境与整体技术栈

5.1.1 开发环境配置

为保证系统在开发与部署阶段的稳定性与可复现性，本文在实际实现过程中对软硬件环境进行了统一规划与配置。主要环境如下：

（1）硬件环境  
本系统主要在实验室工作站环境下开发与测试，典型配置包括：多核 CPU、32GB 以上内存以及具备显存容量的 NVIDIA GPU（用于人物识别模型推理加速）。在无 GPU 条件下，系统亦可退化为 CPU 推理模式，但图像识别延迟会有所增加。

（2）操作系统与基础软件  
开发环境基于 Windows 10/11 与类 Linux 环境混合使用。后端与模型服务主要在 Python 虚拟环境中运行，前端则依赖 Node.js 生态进行构建。具体配置包括：  
① Python 3.10+ 与虚拟环境（venv 或 conda），用于安装 FastAPI、PyTorch、YOLO 等依赖库；  
② Node.js 与 pnpm/npm，用于管理 Vue 3 前端项目依赖并执行构建命令；  
③ SQLite 作为轻量级本地数据库，用于存储任务记录与配置等数据；  
④ 浏览器（Chromium 内核）用于前端调试与交互测试。

（3）第三方服务配置  
系统涉及高德地图 Driving API、DeepSeek Chat API 与 EmailJS 邮件服务等外部资源。为保证安全性，相关密钥与访问令牌均通过环境变量或配置文件的方式进行管理，不在代码仓库中直接明文存储。前端在调用 DeepSeek 与 EmailJS 时，采用其官方提供的 HTTP 接口与 SDK，并在配置中限定允许来源域名。

5.1.2 技术栈与项目结构

结合项目需求与开发经验，系统采用“前端富交互 + 轻量后端服务 + 本地/云端模型”的技术路线。主要技术栈包括：

（1）前端技术栈  
前端基于 Vue 3 + TypeScript + Vite 构建，配合 Tailwind CSS、Element Plus 等组件库实现界面布局与交互控件。前端项目主要目录结构如下：

- `src/views/PathPlanningPage.vue`：智程导航功能介绍页面；  
- `src/views/RoutePlanningView.vue`：路径规划主界面；  
- `src/views/PersonRecognitionPage.vue`：智眸千析功能介绍页面；  
- `src/components/PersonRecognitionApp.vue`：人物识别应用主组件；  
- `src/views/ContactView.vue`：智能咨询与自动邮件页面；  
- 其他如 `ImageUploader.vue`、`AnalysisResults.vue`、`AnalysisAssistant.vue` 等复用组件。

（2）后端技术栈  
后端采用 FastAPI 作为 Web 框架，通过 Uvicorn 启动 ASGI 服务。核心目录包括：  
- `app/routers/route_planning.py`：路径规划与高德 API 调用路由；  
- `app/routers/chat.py`：统一聊天与任务路由，实现根据 content.type 分发；  
- `app/routers/image_recognition.py`：图像上传与人物识别接口；  
- `app/routers/mock_image_recognition.py`：人物识别 Mock 接口，便于前端联调；  
- `app/routers/image_analysis_chat.py`：图像识别结果与 LLM 的交互接口；  
- `app/utils/video_analyzer.py`、`app/models/*` 等工具与模型封装模块。

（3）算法与模型层  
算法模型层主要包括本地部署的大规模语言模型（通过 Ollama 或类似接口调用）、YOLO 系列检测模型、ResNet50 年龄估计模型以及自定义衣物颜色识别模块。通过统一的封装接口，后端服务可以在不关心模型内部细节的情况下完成推理调用。

5.2 前端实现

5.2.1 智程导航界面实现

（1）界面布局与组件划分  
智程导航模块的主界面由 `RoutePlanningView.vue` 实现，其整体布局遵循“左侧控制面板 + 右侧地图区域”的设计模式。左侧面板包括自然语言输入区域、起终点与途经点配置区域、多策略选项与路线列表；右侧嵌入地图组件，用于展示规划路径、路况信息与关键节点标注。  
在组件划分上，路径规划界面通过组合多个子组件实现：例如负责输入与按钮交互的表单组件、用于展示推荐路线的列表组件以及负责地图渲染的地图组件，将复杂界面拆解为多个职责单一的可复用模块。

（2）与后端的交互逻辑  
前端通过统一的状态管理（如 Pinia 的 `useMainStore`）封装请求逻辑，用户在自然语言输入框中描述出行需求时，界面会构造包含 content.type 与具体文本内容的请求体，通过 `/chat/completions` 或 `/plan` 接口发送至后端。  
当用户选择以表单方式手动配置起点、终点与途经点时，界面会绕过大模型步骤，直接构造高德路径规划所需参数并提交后端，后端则仅负责与高德 API 通信与结果封装。

（3）路线推荐展示与交互细节  
后端返回的 `RouteResponse` 结构中包含多条候选路线的时间、距离、费用等信息。前端在接收到响应后，会将其映射为可视化的路线列表与地图覆盖物：  
① 在列表中以卡片形式展示每条路线的关键指标，并用“推荐路线”“备选路线”等标签区分；  
② 在地图上以不同颜色与线型绘制各条路径，支持鼠标悬停高亮与点击切换当前选中路线；  
③ 在侧边栏展示详细路段信息，如途经道路名称、转向提示、途经城市等，帮助用户理解路线细节。  
同时，系统为请求过程提供加载状态与错误提示，例如在请求超时或高德 API 返回异常时，在界面上以消息框形式进行友好反馈。

5.2.2 智眸千析界面实现

（1）图像上传与预览交互  
人物识别应用由 `PersonRecognitionApp.vue` 负责实现，配合 `ImageUploader.vue` 等组件支持拖拽上传、点击选择文件等多种方式。上传成功后，界面会立即在预览区域展示原始图像，并在用户确认后触发后端识别请求。为避免过大文件导致延迟，前端在上传前对图像类型与大小进行基本校验，并在必要时进行压缩或尺寸限制提示。

（2）识别结果展示与属性标注  
后端返回的人物识别结果采用统一的 `persons` 结构，其中包含每个人物的边界框坐标、年龄段、性别、上衣颜色、下衣颜色及对应置信度。前端根据该结构在图像上绘制矩形框，并在框旁展示属性标签。  
为提升可读性，系统在界面上引入颜色图例与图标：例如以不同颜色的标签表示性别、以文字标注年龄段与服装颜色。当同时存在多个人物时，界面允许用户在侧边栏中按编号选择某个人物，以高亮显示对应边界框与详细属性信息。

（3）与图像分析聊天模块的集成  
在需要进一步解释识别结果或进行按描述检索时，用户可以在 `AnalysisAssistant.vue` 中输入自然语言问题。前端会将当前识别结果与问题文本一并提交至 `/image_analysis_chat` 接口，由 LLM 生成解释性回答。  
生成的回答展示在对话区域，用户可以多轮提问，以获得针对特定人物或整体场景的分析结果，例如“哪些人穿红色上衣？”“图中是否有可疑人员特征？”等，从而实现图像识别与自然语言问答的联动。

5.2.3 联系与自动邮件界面实现

5.2.4 智航监控与数据大屏界面实现

（1）数据监控总览与导航入口  
智航监控相关前端主要以数据大屏形式呈现，通过导航栏入口直接跳转到监控视图或功能特性页。监控视图中，上方为标签式导航（总览、视频监控、状态监控、数据分析等），下方以网格布局组合无人机状态面板、视频监控组件、地图追踪组件与数据图表等卡片式子视图。该设计一方面保持了各监控子模块的相对独立性，另一方面通过统一的布局与配色风格，形成“飞行状态—视频画面—空间位置—统计图表”的一体化监控界面。

（2）设备状态与无人机巡逻面板实现  
设备状态组件通过模拟或真实接口获取各类监控设备与无人机的属性信息，并在前端维护统一的状态数组。组件内部对设备进行类型与状态筛选，在卡片中展示名称、健康度进度条、位置与状态指示灯，并提供分页与每页条数调节等功能，以适应大规模设备集合的浏览需求。无人机巡逻面板则提供任务配置向导式界面：包括巡逻地点选取、天气信息查询、无人机智能推荐、任务类型选择与巡逻区域绘制等步骤，并在任务运行阶段以动态列表的形式展示各无人机的电量、信号、状态与任务进度，同时集成简化的视频监控面板，用于查看特定无人机的模拟视频画面。

（3）视频监控分屏与交互效果  
视频监控组件采用可切换的单屏/多屏视图模式，内部维护一个包含多路“无人机视频源”的列表，每个视频源包含标识、场景类型、位置、在线状态与当前告警级别等字段。组件通过对该列表的过滤与分页，实现按监控类型（如车牌识别、人物识别、火灾监测等）与在线状态的筛选显示；在视觉层面，使用图片资源模拟实时视频流，并在其上叠加代表车牌检测框、人物框、火点区域等的图形元素，以体现不同监控类型下的语义信息。为增强临场感，组件还引入轻量动画效果（如抖动、告警闪烁等）模拟无人机视角抖动与告警高亮。

（1）表单设计与校验  
`ContactView.vue` 实现了智能咨询与自动邮件模块的前端界面。表单包括姓名、邮箱、电话、公司、微信号等基础信息字段，以及复选或下拉形式的“感兴趣功能类型”和多行文本的“需求描述”。  
在实现上，表单组件结合 Element Plus 的表单验证机制，对必填项与邮箱格式等进行校验，并在用户提交前给出错误提示，避免空表或格式明显错误的内容直接发送。

（2）调用大模型生成咨询摘要  
当用户点击“提交咨询”时，前端首先调用封装好的 `processMessageWithDeepSeek` 方法。该方法会将项目能力说明与用户填写的内容组合为完整 Prompt，发送至 DeepSeek Chat API。  
接收到模型返回的文本后，前端通过 `formatProcessedContent` 等解析函数，从中提取标题、大纲、建议等结构化字段。若模型返回格式不符合预期，则自动触发 `formatFallbackResponse`，生成一个内容简洁但结构完整的基础摘要，以保证后续邮件渲染的稳定性。

（3）邮件模板渲染与发送反馈  
在获得结构化摘要与用户信息后，前端构造 EmailJS 所需的模板参数，将标题、摘要内容、建议与原始留言等字段映射到预设 HTML 模板中。随后调用 EmailJS SDK 的发送函数，将邮件发送至预设接收邮箱。  
发送过程中，界面会显示加载状态并在发送成功或失败后给出明确提示。对于失败情况，系统提示用户检查网络或稍后重试，并可在控制台输出错误信息以便开发与运维人员排查。

5.3 后端实现

5.3.1 路径规划与聊天统一入口实现

（1）`route_planning.py` 路由实现  
路径规划服务通过 `app/routers/route_planning.py` 对外提供 `/plan` 等接口。该路由负责接收前端提交的结构化规划参数或自然语言需求，根据入参类型选择不同的处理路径：  
① 若已包含起终点与途经点等结构化字段，则直接调用高德 API 进行多策略路径搜索；  
② 若仅包含自然语言描述，则调用本地 LLM 进行语义解析，得到结构化参数后再执行高德调用。  
路由在调用高德 API 前会对参数进行合法性检查，并在调用后将返回的路线信息转换为内部统一的 `RouteResponse` 数据结构，以 JSON 形式返回前端。

（2）`chat.py` 统一聊天与任务路由  
为避免前端针对不同任务编写多套接口调用逻辑，后端在 `app/routers/chat.py` 中实现了统一聊天路由 `/chat/completions`。该路由根据请求体中的 content.type 字段判断当前任务类型，如“route_planning”“image_analysis”等，并将请求分发到相应的处理函数（如 `handle_route_planning`）。  
在路径规划场景下，`handle_route_planning` 会构造 ROUTE_PROMPT，将用户自然语言与必要上下文组合后提交给本地 LLM，解析得到结构化参数，再调用路线规划服务完成后续处理。该设计使得系统在接口层呈现出统一的“对话式”交互风格，便于前端扩展与维护。

（3）错误处理与日志记录  
在路径规划与聊天路由实现中，系统对外部服务调用过程中的异常进行了统一捕获与处理。例如在高德 API 请求超时、LLM 返回错误或解析失败时，路由会返回包含错误码与简要说明的响应体，并在后端日志中记录详细的异常堆栈信息。  
同时，系统可在数据库中记录关键请求与响应摘要，为后续性能分析与问题排查提供依据。

5.3.2 图像识别与多模型推理服务实现

（1）`image_recognition.py` 图像识别接口  
人物识别服务通过 `app/routers/image_recognition.py` 暴露 `/image-recognition` 等接口，接收前端上传的图像文件。接口实现流程包括：  
① 对上传文件进行类型与大小校验，并在服务器端保存至临时目录；  
② 调用封装好的图像处理管线（如 `ImageProcessor`），依次执行 YOLO 检测、年龄性别识别与服装颜色分析；  
③ 将每个人物的识别结果封装为统一的 `persons` 列表，并以 JSON 格式返回前端。

（2）模型加载与推理优化  
为减少重复加载带来的性能开销，系统在服务启动阶段预加载 YOLO 模型与 ResNet 年龄估计模型，并将其保存在全局或依赖注入容器中，供后续请求复用。在推理过程中，根据硬件环境选择 GPU 或 CPU 设备，并在必要时对输入图像进行尺寸缩放与批量处理，以平衡推理速度与识别精度。  
对于推理耗时较长的场景，系统还可以通过异步任务或队列机制进行扩展，但在当前实现中，以同步方式返回结果即可满足一般实验与演示需求。

（3）`mock_image_recognition.py` 模拟服务  
为了在后端模型尚未就绪或推理环境受限时仍能进行前端联调，系统实现了 `mock_image_recognition.py` 路由。该路由不调用真实模型，而是直接返回预定义或随机生成的 `persons` 结构数据，使前端能够在开发早期完成界面布局与交互逻辑的测试。  
通过在前端配置中切换真实接口与 mock 接口，开发者可以灵活选择是否启用真实推理服务，从而提高开发效率与系统的可演示性。

5.3.3 图像分析聊天与多模态接口实现

（1）`image_analysis_chat.py` 路由设计  
多模态分析服务由 `app/routers/image_analysis_chat.py` 提供，主要处理“识别结果 + 用户问题”的联合输入。该路由接收前端传来的 `persons` 结构与文本问题，将其组织为适合大模型理解的 Prompt，例如将每个人物的属性以列表形式嵌入，同时附加用户提出的问题。  
随后，路由调用本地或云端大模型接口获取自然语言回答，并将回答文本返回前端展示。

（2）Prompt 构造与结果封装  
在 Prompt 设计上，系统将人物属性结构化信息转换为简洁的自然语言描述，例如“编号 1：男性，青年，红色上衣，黑色长裤”等，并在此基础上附加用户问题，如“图中是否有穿红色上衣的人？”。通过这种方式，模型能够在统一文本上下文中理解图像识别结果与用户意图。  
返回结果以简单的文本字段封装，前端可以直接在对话窗口中展示，或根据需要进一步解析出结构化信息。

5.3.4 智航监控与车牌监控服务实现

（1）车牌监控路由设计  
在后端服务中，智航监控通过专用的车牌监控路由实现对目标车辆的识别与告警。该路由首先提供“目标车牌设置”接口，接收用户上传的车牌图片或直接输入的车牌号，并通过一次离线识别得到候选车牌集合，最终将用户确认的目标车牌及其颜色缓存为监控状态。随后，“视频上传与分析”接口接收监控视频文件，对视频进行分帧与降采样处理，在每个采样帧上调用车牌检测与识别管线，输出车牌文本、颜色信息、边界框坐标与置信度等字段，并与缓存的目标车牌进行相似度匹配；匹配成功的帧将被记录为“关键帧”，用于后续可视化与告警。

（2）识别管线集成与结果持久化  
车牌监控路由内部集成了项目中已有的车牌检测/识别与车辆颜色识别模型，并在此基础上增加了结果后处理与多源输出功能。一方面，对同一视频中多帧的识别结果进行去重与排序，将所有唯一车牌号写入文本文件，便于人工复核与统计分析；另一方面，将匹配目标车牌的帧按照时间顺序编码为短视频，并在帧图像上叠加车牌框与文字标注，用于在前端或离线报告中展示。此外，路由还维护每个处理任务的状态与统计信息（总帧数、处理帧数、匹配次数等），以 JSON 形式持久化到结果文件中，为后续性能分析与系统调优提供数据基础。

（3）告警通知与邮件联动  
为实现从视频分析到运营响应的闭环，车牌监控服务在检测到目标车牌或处理过程中出现异常时，会构造告警摘要（包括处理任务 ID、时间戳、目标车牌号、匹配帧截图路径等），并通过邮件通知模块发送至预设邮箱。邮件中除文字说明外，还可附加关键帧截图或匹配视频链接，便于运维人员快速了解告警背景。对于手动触发的报警需求，路由还提供独立的“手动报警接口”，支持在无需视频分析的情况下直接发送包含指定车牌号与说明信息的告警邮件。

5.3.5 第三方服务访问与配置管理

（1）高德地图 API 调用封装  
系统在后端实现了对高德 Driving API 的调用封装，包括请求 URL 拼接、参数编码与响应解析等。为适应多策略路径规划需求，封装函数支持根据不同策略参数发起多次请求，并对结果进行合并与去重。  
高德 API 密钥通过环境变量或配置文件加载，避免硬编码在代码中；同时，在调用过程中对 HTTP 状态码与返回体中的错误信息进行检查，并在出现配额限制或其他异常时返回可解释的错误提示。

（2）DeepSeek Chat API 调用  
尽管智能咨询模块主要在前端直接调用 DeepSeek API，但后端仍可预留代理接口，以便在未来统一管理访问密钥与请求限流。当前实现中，前端通过 HTTPS 向 DeepSeek 服务发送请求，并在响应中获取生成的摘要文本。  
在调用参数设置上，系统适当控制 temperature、top_p 等生成超参数，并限制最大生成长度，以提高摘要结果的稳定性与可解析性。

（3）EmailJS 集成与配置  
EmailJS 的服务 ID、模板 ID 与公钥同样通过配置项管理。前端在发送邮件时仅使用这些标识与模板变量进行调用，具体 SMTP 服务器与投递细节由 EmailJS 服务端负责处理。  
为防止滥用与垃圾邮件风险，系统可在前端增加基础防护措施，如频率限制与验证码等，论文层面则重点说明其集成方式与配置管理策略。

5.4 系统部署与运行

5.4.1 本地开发与调试流程

在本地开发环境中，系统的启动与调试流程大致如下：  
（1）后端：在 Python 虚拟环境中安装依赖后，通过 `uvicorn app.main:app --reload` 启动 FastAPI 服务；  
（2）前端：在 Node.js 环境中执行 `pnpm install`（或 `npm install`）安装依赖，再通过 `pnpm dev` 启动 Vite 开发服务器；  
（3）模型服务：根据需要启动本地 LLM 服务（如 Ollama）与加载 YOLO/ResNet 模型，确保相关端口与路径配置正确；  
（4）第三方服务：在 .env 或配置文件中正确填入高德 API Key、DeepSeek API Key 与 EmailJS 配置信息。  
前端通过代理或 CORS 配置与后端交互，开发者可在浏览器中直接访问前端地址进行功能测试。

5.4.2 部署方案与运行策略

在部署阶段，系统可根据实际资源条件选择单机或多机部署方案。典型方案为：在一台服务器上同时部署前端静态资源与后端 API 服务，并在同机或专用机器上运行模型推理服务。  
前端构建完成后，可将生成的静态文件部署到 Nginx 等 Web 服务器上；后端通过 Uvicorn + Nginx 反向代理的方式对外提供统一的 API 入口。本地 LLM 服务与视觉模型推理进程根据需要设置为系统服务或使用进程管理工具（如 Supervisor）进行守护。  
在运行策略上，系统通过日志记录、异常捕获与基础监控等手段，跟踪各模块的运行状态与性能指标，为后续的优化与扩展提供数据支持。

5.5 本章小结

本章在第四章核心算法设计的基础上，从工程实现角度详细介绍了系统的开发环境、前后端实现细节以及与第三方服务的集成方式。前端部分重点阐述了智程导航、智眸千析与智能咨询界面的布局与交互逻辑；后端部分则说明了路径规划、图像识别、多模态分析与统一聊天路由的实现过程，并对高德、DeepSeek、EmailJS 等服务的访问与配置管理进行了说明。最后通过对本地开发与部署流程的介绍，为后续实验测试与性能评估提供了工程基础。
