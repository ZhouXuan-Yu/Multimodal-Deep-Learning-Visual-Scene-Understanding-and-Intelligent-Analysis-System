二、相关技术与理论基础

2.1 无人机路径规划与地图服务基础

2.1.1 路径规划问题定义

路径规划（Path Planning）是无人机、移动机器人与智能车辆等系统中的核心问题之一，其目标是在给定环境模型、起点与终点的条件下，求解一条或多条满足约束条件的可行路径，并在安全性、时间、距离、能耗或成本等指标上实现优化。形式上，可将路径规划抽象为带约束的图搜索或优化问题：在状态空间或图结构上，从起始节点出发，寻找到终止节点的最优或次优路径，并在搜索过程中规避障碍区域、风险区域等不可行区域。

设道路网络可表示为带权有向图 $G=(V,E)$，其中 $V$ 为路网节点集合，$E$ 为边集合，每条边 $e=(u,v)\in E$ 关联有代价函数 $w(e)$，可综合反映时间、距离、费用等因素。给定起点 $s\in V$ 与终点 $t\in V$，路径规划问题可形式化为在所有从 $s$ 到 $t$ 的可行路径集合 $\mathcal{P}(s,t)$ 中求解
$$
P^\* = \arg\min_{P\in \mathcal{P}(s,t)} J(P),
$$
其中 $J(P)$ 为对路径综合代价的评价函数，可为单目标或多目标加权形式。

在无人机巡检与安防场景中，路径规划不仅要考虑地理距离与时间成本，还需要结合业务需求设置巡逻点、重点关注区域以及避让区域等。例如，小区巡逻需要覆盖多个出入口与公共区域，园区巡检则需围绕关键基础设施布设巡逻路线。因此，路径规划往往表现为多目标、多约束的组合优化问题，需要在规划效率与方案质量之间取得平衡。

2.1.2 典型路径规划算法

传统路径规划算法主要包括基于图搜索的方法与基于智能优化的方法两大类。  
基于图搜索的方法通过将环境离散为网格或图结构，在此基础上采用启发式或非启发式算法进行搜索。例如，Dijkstra 算法是一种经典的单源最短路径算法，可在非负权重图上计算从起点到各节点的最短路径；A* 算法在此基础上引入启发式函数，对搜索方向进行引导，在保证找到最优解的前提下有效缩小搜索空间，常被用于网格地图与道路网络的路径规划。

从时间复杂度角度看，在稠密图上，经典 Dijkstra 算法基于优先队列的实现复杂度为 $O(|E|\log |V|)$，A\* 算法在采用一致、可接受的启发式函数时仍可保证最优性，且在实际路网中往往能显著减少实际访问节点数，提高搜索效率。  
基于智能优化的方法则包括遗传算法（GA）、粒子群优化（PSO）、蚁群算法（ACO）等，这类算法通过模拟自然界群体行为或进化机制，在连续或离散空间中进行全局搜索，适用于具有复杂约束、非线性代价函数的规划问题。其优势在于全局搜索能力较强，但对参数设置较为敏感，且在实时性要求较高的在线规划中存在一定局限，通常更适合离线路径生成或作为高层策略优化手段。

在实际工程系统中，往往会将商用地图服务提供的路线规划结果与本地算法结合使用。一方面，利用地图服务 API（如驾车、步行、骑行路线）快速获取符合道路网络约束的候选路径；另一方面，在此基础上叠加业务层约束或采用算法进行二次优化，以适应具体场景需求。

2.1.3 高德地图服务与路径规划接口

高德地图是国内应用广泛的地图服务提供商之一，其开放平台提供了多种 Web 与移动端 API，包括地图展示、地点搜索、路线规划等功能。对于本系统而言，高德地图的驾车路线规划（Driving）服务尤为关键。通过向高德服务端提交起点、终点、途经点等参数，以及所选策略（如最短时间、最短距离、避开高速等），可以获得包含多条候选路线的规划结果，每条路线包括距离、时间、路段信息与坐标序列等数据。

在实际应用中，系统前端通过 JavaScript SDK 加载高德地图并进行地图渲染；后端负责调用高德路线规划 API，将从大模型解析得到的结构化地址或坐标转换为请求参数，获取规划结果后再返回前端进行可视化展示。通过将语言模型、地图服务与前端交互结合，可实现“用户自然语言描述—参数解析—路径规划—地图可视化”的完整闭环。

2.2 计算机视觉与人物识别技术

2.2.1 目标检测基础

目标检测（Object Detection）是计算机视觉领域的基础任务之一，其目标是在图像或视频中同时完成目标类别的识别与目标位置的定位。典型方法可分为两阶段方法与单阶段方法两类。两阶段方法以 R-CNN 系列为代表，通过候选区域生成与区域分类两步完成检测，具有较高精度但推理速度相对较慢；单阶段方法以 YOLO、SSD 等为代表，将检测任务看作密集预测问题，直接在特征图上回归边界框和类别概率，具有更高的实时性，适合对推理速度要求较高的监控与无人机应用场景。

YOLO（You Only Look Once）系列算法在保持较高检测精度的同时，极大提升了推理速度，被广泛应用于工业场景。其核心思想是在单个网络中同时完成特征提取、边界框回归与类别预测，通过网格划分与多尺度特征融合处理不同尺寸目标，从而在实时视频流中实现高效检测。

2.2.2 人物属性识别与多特征分析

在安防与监控场景中，仅检测到“有人”还不足以支撑复杂任务，往往需要进一步识别人物的多种属性信息，例如年龄段、性别、服装颜色与穿着风格等。这类任务通常被统称为人物属性识别（Person Attribute Recognition）。在技术实现上，一种常见思路是基于检测到的行人框，使用专门的分类或回归模型对属性进行预测；另一种思路则是将行人再识别（Re-ID）与属性识别结合，通过共享特征表示同时完成身份区分与属性估计。

深度卷积神经网络（如 ResNet、DenseNet 等）已被证明在人物属性识别任务上具有较好的性能。通过在大规模标注数据集上训练，这类网络能够学习到与性别、年龄段、服饰风格等相关的中高层语义特征。对于颜色识别，则可以结合传统颜色空间转换（如 RGB 到 HSV）与深度网络特征，对上衣、下装等区域的主色调进行分类，从而得到诸如“红色上衣”“黑色长裤”等语义描述。

在评价指标方面，人物属性识别通常采用分类准确率（Accuracy）、精确率（Precision）、召回率（Recall）与 F1 值等指标刻画分类性能；对于年龄估计等回归或有序分类任务，还常采用平均绝对误差（Mean Absolute Error, MAE）或均方误差（MSE）衡量预测值与标注值的偏差。在后续实验章节中，本文将参考上述指标对不同属性识别子模块的效果进行定量评估。

2.2.3 多模型融合与系统接口设计

在实际工程系统中，单一模型往往难以满足复杂场景下的鲁棒性需求。为此，可以采用多模型融合与结果后处理策略，对人物属性识别结果进行增强。例如，可将行人检测模型与专门的年龄性别识别模型、服装颜色识别模型组合使用：首先由检测模型输出人物区域，再分别送入不同属性模型进行推理，最后在业务逻辑层对多个属性结果进行统一编码与校验。

在接口设计上，系统需要将多模型输出的属性结果封装为统一的结构化数据格式，包括人物编号、年龄段、性别、上衣颜色、下装颜色等字段，并附带置信度信息，以便前端展示与后续检索使用。这种统一的接口设计，不仅方便前端以表格或标签形式直观展示识别结果，也为未来扩展更多属性（例如是否佩戴帽子、背包等）预留了空间。

2.3 大规模语言模型与 Prompt 设计

2.3.1 大规模语言模型概述

大规模语言模型通过在海量语料上预训练，学习自然语言中的统计规律与语义关系，在文本生成、问答、翻译、推理等任务上表现出强大的泛化能力。典型模型包括 GPT 系列、LLaMA 系列、国内的 DeepSeek 等。这些模型通常采用 Transformer 架构，通过自注意力机制建模序列中任意位置的依赖关系，从而在长文本建模与复杂语义推理上具有优势。

在工程应用中，大模型既可以作为云端服务调用，也可以在本地部署轻量化版本，以兼顾隐私、安全与响应速度等需求。对于本系统而言，本地部署大模型主要用于对用户自然语言需求进行解析与结构化转换，使其成为路径规划、人物识别与邮件服务等功能模块的“上游语义入口”。

2.3.2 Prompt 设计与参数结构化抽取

为了让大模型在具体业务场景中发挥稳定可靠的效果，合理的 Prompt 设计十分关键。Prompt 设计通常包括角色设定、任务描述、输入输出格式约束与示例（few-shot）等部分。例如，在路径规划场景中，可以将大模型设定为“出行规划助手”，明确要求其将用户自然语言描述解析为起点、终点、多个途经点、出行偏好等字段，并以 JSON 或特定键值对格式输出。通过在 Prompt 中加入若干示例，模型可以更好地学习到所需结构化输出的模式。

在人物识别和智能咨询场景中，Prompt 设计同样重要。前者可以通过描述“你将接收到来自视觉模块的属性标签，请根据这些标签生成简洁自然的中文描述”等约束，引导模型进行自然语言生成；后者则需要引导模型对用户的咨询内容进行摘要、分类与优先级判断，并提取出关键信息如联系方式、问题类型等，为自动邮件发送提供结构化输入。

2.3.3 本地部署与调用方式

本地部署大模型通常需要考虑模型大小、显存占用、推理速度与并发能力等因素。常见做法是采用量化模型或轻量版本，通过本地推理引擎（如基于 C++ 或 Python 的推理框架）对外暴露 HTTP 或 WebSocket 接口。应用系统通过调用本地推理接口，将用户输入与 Prompt 拼接后发送至模型，并对返回结果进行解析与后处理。

在本系统中，本地大模型主要承担两类任务：一是将用户自然语言需求转化为结构化参数，例如路径规划请求中的位置与偏好信息；二是对用户填写的表单与留言内容进行归纳与改写，为自动邮件发送提供高质量文本。这种“本地模型 + 业务服务”的集成方式，有助于在保证数据安全的前提下，提供低延迟、可控性强的智能服务能力。

2.4 邮件服务与系统集成技术

2.4.1 邮件发送机制与常用协议

电子邮件（E-mail）是信息系统中最常用的异步沟通方式之一。传统邮件发送通常采用 SMTP（Simple Mail Transfer Protocol）协议，由客户端将邮件投递到邮件服务器，再通过多级转发最终送达收件人。现代 Web 应用中，开发者常借助第三方邮件服务或封装库，简化 SMTP 配置与身份验证流程，使邮件发送像调用普通 HTTP 接口一样简单。

在实际工程实践中，邮件服务需要支持身份认证、SSL/TLS 加密、模板渲染与批量发送等功能，以保证安全性与灵活性。同时，合理的邮件主题与内容格式设计，有助于收件人在众多邮件中快速定位关键信息，提高沟通效率。

2.4.2 EmailJS 等第三方服务

EmailJS 是一种面向前端开发者的邮件发送服务，可通过 JavaScript 在浏览器端直接调用，无需在本地维护邮件服务器。开发者只需在 EmailJS 平台配置服务 ID、模板 ID 与用户密钥，即可在前端代码中调用其 API，将表单数据与自定义内容注入预先配置好的邮件模板，实现自动邮件发送。

对于本系统而言，EmailJS 等服务的引入可以大幅降低邮件模块的开发与运维成本，使系统能够在前端就完成表单收集、内容生成与邮件发送的完整流程。同时，结合大模型对咨询内容的摘要与重写能力，可以在发送前自动优化邮件内容，使其更符合正式沟通场景的表达规范。

2.4.3 前后端集成与安全性考虑

在系统集成层面，前端负责采集用户输入、调用本地大模型或后端接口生成摘要内容，并向邮件服务提交发送请求；后端则可根据需要对部分敏感信息进行校验与存储，例如记录用户咨询历史与系统反馈状态等。在调用第三方服务时，需要妥善管理密钥与配置，避免在公开代码仓库中泄露敏感信息，可采用环境变量或后端代理等方式进行保护。

同时，为防止恶意滥用邮件发送功能，应在业务逻辑中加入频率限制与简单的人机验证机制，例如限制同一用户或同一 IP 在一定时间内的发送次数，以保证系统的稳定性与服务质量。
