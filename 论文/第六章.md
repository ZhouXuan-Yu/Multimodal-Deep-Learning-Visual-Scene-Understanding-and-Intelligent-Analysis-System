六、实验结果与分析

6.1 实验环境与数据集说明

6.1.1 实验环境配置

为评估本系统在实际应用场景中的性能与效果，本文在前述开发环境的基础上，对实验硬件与软件环境进行了统一配置。主要环境如下：

（1）硬件环境  
实验主要在实验室工作站上完成，配置包括：多核 CPU、32GB 内存以及一块支持 CUDA 的 NVIDIA GPU，用于加速人物识别模型推理。部分对比实验在无 GPU 条件下进行，以测试在资源受限场景下的性能变化。

（2）软件环境  
后端运行在 Python 3.10 环境中，集成 FastAPI、Uvicorn、PyTorch、Ultralytics YOLO 等库；前端基于 Vue 3、TypeScript 与 Vite 搭建，通过现代浏览器访问；本地 LLM 服务通过 Ollama 或同类框架部署，DeepSeek Chat 与高德地图 Driving API 通过 HTTPS 接口调用。  
所有实验均在相同版本的依赖与配置下完成，以保证对比结果的可重复性。

6.1.2 数据集与测试场景

（1）路径规划场景数据  
为评估智程导航模块的路径规划效果，本文选择若干典型城市路网场景，构造了 30 组包含起点、终点与不同出行偏好的测试用例，涵盖日常通勤、跨区出行与多途经点任务等多种情况。  
在部分用例中，输入形式为结构化参数；在另一些用例中，输入形式为自然语言描述，用于评估大模型语义解析对结果的影响。

（2）人物识别数据  
智眸千析模块的实验数据来自公开行人/人脸数据集与自采集监控场景图像。为了更贴近安防与巡检需求，本文选取了室内、室外、白天、夜间等多种光照与场景条件，共计 500 张图像，包含不同年龄段、性别与服装颜色组合的人物。  
在标注方面，采用人工标注的年龄段、性别与服饰颜色作为“伪真值”，用于计算识别精度与误差指标。

（3）智能咨询与自动邮件测试场景  
智能咨询模块的实验主要基于模拟与真实的用户咨询案例。本文构造了 20 份覆盖不同需求类型的咨询表单，包括路径规划需求、人物识别需求、系统集成需求等，并邀请若干具有工程背景的同学/老师对摘要质量与可读性进行主观评价。

6.2 智程导航模块实验与分析

6.2.1 路径规划有效性分析

为评估 LLM + 高德组合的路径规划效果，本文将“基于语义解析的多策略规划”与“仅使用高德默认规划策略”进行对比。对每组测试用例，分别记录两种方案下的推荐路线，并从以下几个方面进行分析：

（1）路线合理性  
通过人工判读与对照高德 App 结果，统计两种方案在“绕路程度”“是否符合出行偏好”等维度上的表现。结果表明，在明确提出“避开收费”或“尽量避开拥堵”的需求时，语义解析方案能够更稳定地生成符合偏好的路线，而默认策略在部分场景下仍会选择收费高速或拥堵路段。

（2）多策略覆盖度  
在 LLM 辅助下，系统会为单次需求生成多条具有不同侧重点的候选路线（时间优先、距离优先、费用优先等），供用户选择。在 30 组测试用例中，绝大多数情况下用户都能在推荐集合中找到与自身偏好高度匹配的路线，相比仅返回单条结果的方案，在决策灵活性上有明显提升。

（3）客观指标对比  
在不改变起终点的前提下，对比两种方案下最终被选中路线的行驶时间、距离与收费等指标。总体来看，LLM + 高德方案在时间与距离指标上与默认方案接近，但在满足用户个性化偏好的前提下，能够在收费或绕行方面取得更优平衡。

6.2.4 对比实验与统计分析建议

在条件允许的情况下，可进一步设计更系统的对比实验与统计检验。例如：  
（1）在相同起终点与约束条件下，对比“LLM + 高德”“仅高德默认策略”“基于规则的参数映射”三种方案的平均行驶时间、距离与费用，并绘制箱线图或累计分布曲线，从而刻画不同方案在多维指标上的差异；  
（2）通过主客观结合的方式评估路线合理性：主观上邀请多名被试对路线满意度进行 Likert 量表打分，客观上统计偏离理论最优解的相对误差；  
（3）利用配对样本 $t$ 检验或非参数检验，判断 LLM 语义解析方案在满足用户偏好方面相对于基线方法的改进是否具有统计显著性。

6.2.2 响应时间与性能测试

除了路线合理性外，路径规划模块的响应时间也是衡量系统可用性的重要指标。本文分别统计了在不同并发请求数量（1、5、10）下，路径规划接口的平均响应时间与成功率。

实验结果表明：在单用户或低并发场景下，路径规划请求的整体延迟主要由高德 API 与本地 LLM 推理耗时构成，总响应时间能稳定控制在数秒范围内；在并发请求数增加时，若本地 LLM 未启用并行推理与缓存机制，则响应时间会有一定增长，但仍处于可接受范围。  
通过对日志与监控数据的分析，可以发现合理设置 LLM 输出长度限制、开启结果缓存等策略，有助于在保证结果质量的前提下进一步降低平均响应时间。

6.2.3 语义解析稳定性分析

在自然语言输入场景下，语义解析的稳定性直接影响路径规划结果。本文在 20 组自然语言用例上进行了多次重复调用，对比同一需求在不同调用中的解析结果差异。

实验表明：在引入 ROUTE\_PROMPT 与少量示例的条件下，大模型能够在大部分用例中输出结构与字段齐全的 JSON 结果，起终点与核心偏好解析较为稳定；个别用例中，会出现输出格式略有偏差或个别字段缺失的情况，但通过后端的格式校验与回退策略（如重新请求或采用默认偏好），可以有效避免系统异常。  
总体而言，语义解析模块在毕业设计规模的实验场景下具备较好的可用性，为路径规划功能提供了较友好的自然语言入口。

6.3 智眸千析模块实验与分析

6.3.1 人物检测与属性识别精度

为评估人物多特征识别模块的精度，本文在前述 500 张测试图像上，分别统计行人检测、年龄段识别、性别识别与服饰颜色识别的性能指标。

（1）行人检测性能  
采用常见的检测评价指标（如 Precision、Recall 或 mAP）对 YOLO 检测结果进行统计。实验结果显示，在适当设置置信度与 NMS 阈值后，系统在中等分辨率图像上的行人检测召回率较高，漏检率较低，能够满足后续属性识别的输入需求。

（2）年龄段与性别识别性能  
将模型输出与人工标注结果对比，统计年龄段分类的准确率与平均绝对误差（MAE），性别识别的分类准确率等指标。结果表明，在光照条件良好、人物朝向较正的图像中，年龄段识别与性别识别均取得较高准确率；在背光、强遮挡或极端角度的情况下，性能会有所下降，但整体仍具备一定鲁棒性。

（3）服饰颜色识别性能  
通过对比自动识别的上衣/下衣颜色标签与人工标注结果，统计颜色识别的准确率。实验发现，在颜色对比明显、光照均匀的场景中，基于 HSV 直方图的主色调识别方法效果较好；在光照复杂或衣物图案较多的情况下，主色调可能与主观感知存在一定偏差，但仍能为“按描述检索人物”等上层应用提供较有用的特征信息。

6.3.2 推理时间与资源占用

在不同硬件环境下，本文测试了人物识别模块在单张图像上的平均推理时间与资源占用情况。

（1）GPU 环境下  
在 GPU 加速条件下，单张中等分辨率图像的完整处理流程（检测 + 属性识别 + 颜色分析）可在数百毫秒至一秒内完成，能够满足在线场景下的交互需求。多人物场景中，由于共享前向传播开销，整体延迟增长相对有限。

（2）CPU 环境下  
在仅使用 CPU 的情况下，推理时间明显增加，尤其在分辨率较高或人物数量较多的图像上，整体延迟可能达到数秒以上，更适合作为离线分析或低频查询场景。  
总体而言，通过合理选择模型规模与输入分辨率，系统可以在不同资源条件下实现“精度—速度”的平衡。

6.3.3 多模态问答效果分析

在完成基础属性识别后，本文进一步评估了基于 LLM 的多模态问答效果。选取若干包含多人物的图像，设计了例如“图中有多少位穿红色上衣的人？”“是否有老年女性？”等问题，并比较系统回答与人工判断的一致性。

实验结果显示：在属性识别结果较为准确的前提下，大模型能够基于结构化 `persons` 数据给出符合逻辑的回答，并能够处理一定程度的组合条件查询（如同时涉及性别、年龄段与服饰颜色）。当底层识别结果存在误差时，回答质量也会相应受影响，表明多模态链路中各子模块间存在误差传递现象。  
总体上，多模态问答模块在实验场景下验证了“结构化视觉特征 + LLM 推理”的可行性，为后续扩展更复杂的监控分析任务提供了基础。

6.3.4 误差分解与消融实验建议

为了更深入理解人物多特征识别与多模态问答链路中的误差来源，可设计以下消融与误差分解实验：  
（1）在固定检测结果的前提下，分别替换或关闭年龄估计、性别识别与颜色识别子模块，比较对最终问答准确率与召回率的影响，从而量化各属性子模块的重要性；  
（2）在人工提供的“理想属性标签”基础上，仅保留 LLM 问答环节，评估 LLM 在属性完美的理想条件下可达到的上限性能，进而与真实流水线结果对比，估计视觉识别环节引入的性能损失；  
（3）在不同光照、遮挡与视角条件下分别统计检测与属性识别的精度，并绘制误差随场景复杂度变化的趋势曲线，以支持对系统鲁棒性的定量分析。

6.4 智航监控模块实验与分析

6.4.1 设备状态与健康度评估实验

在智航监控模块中，设备状态与健康度评估是支撑后续告警与运维决策的基础。虽然当前实现以模拟数据为主，但仍可在实验层面验证状态建模与可视化设计的合理性。具体而言，可构造若干包含不同类型监控设备与无人机的配置场景，设定温度、电量、信号强度等指标的演化规律，统计在一定时间窗口内各状态类别（normal / warning / critical / offline）的占比变化，并观察状态面板在设备数量增加或状态持续波动情况下的可读性与响应性；同时，可通过人为注入异常（如温度突增、电量骤降）验证健康度阈值与状态平滑策略在避免“误报/漏报”之间的权衡效果。

6.4.2 视频监控场景覆盖与告警可视化

针对视频监控子模块，可以从“场景类型覆盖度”和“告警展示效果”两个角度进行实验评估。一方面，统计系统在多种典型场景（车牌监控、森林火灾、夜间街道、夜间车辆、远距离监控等）下的视频源配置与检测结果模拟情况，分析不同场景下告警频次与告警级别分布是否符合预期；另一方面，从主观角度邀请被试体验单屏/多屏视图切换、检测框叠加、告警闪烁与统计摘要等可视化效果，对“重要事件是否足够突出”“多路监控画面下信息是否过载”等问题进行问卷调查，从而为后续优化视觉编码与交互方式提供依据。

6.4.3 目标车牌监控与报警链路验证

对于车牌监控与告警子模块，更关注的是从目标设定、视频分析到报警通知的整体链路是否可靠。实验上可设计若干包含目标车牌与干扰车牌的视频片段，控制目标车牌出现的次数、位置、清晰度与遮挡程度，统计系统在不同场景下的目标检出率、误报率以及匹配视频/关键帧生成的完整性；同时记录从视频上传到告警邮件送达的端到端时间分布，评估在典型工作负载下链路延迟是否满足实际运维场景的可接受范围。结合这些结果，可以对车牌相似度阈值、帧采样策略与邮件触发条件等参数进行调优。

6.5 智能咨询与自动邮件模块实验与分析

6.5.1 摘要质量与可读性评估

为评价智能咨询模块生成摘要的质量，本文设计了包含 20 份咨询案例的问卷实验。每份案例均提供用户原始留言以及系统生成的摘要内容，并邀请若干评审从“信息完整性”“结构清晰度”“语言表达质量”三个维度进行 1–5 分打分。

实验结果表明：在大多数案例中，系统生成的摘要能够较好覆盖用户提出的关键需求点，并以分条或分段的形式进行组织，结构较为清晰；语言表达在正式程度与可读性方面也能满足工程沟通场景的基本要求。个别案例中，当用户原始描述极为简略或语义模糊时，模型生成的建议部分会显得相对笼统，但整体仍具有参考价值。

6.5.2 结构化解析鲁棒性分析

在生成摘要后，系统需要通过解析算法将自然语言输出映射到 HTML 模板结构。为评估解析算法的鲁棒性，本文对 20 份案例的输出进行分析，统计“完全按预期格式解析”“部分结构识别成功”“启用回退模板”三种情况的占比。

实验表明：在加入合理的输出格式约束与示例后，大多数案例能够被准确解析出标题、大纲与正文部分，仅少数输出因分隔符不统一而需要部分回退。即便在完全启用回退模板的情况下，邮件内容仍保持基本的标题与正文结构，没有出现因解析失败导致内容缺失的情况。  
这表明通过规则与回退策略相结合的方式，可以在不引入复杂解析模型的前提下，较好地提升整体系统的鲁棒性。

6.5.3 工程效率与用户体验分析

在工程实践层面，智能咨询与自动邮件模块主要目标是减少沟通轮次、提升问题响应效率。虽然本毕业设计未进行大规模线上 A/B 测试，但根据小规模使用反馈与模拟场景评估，可以观察到以下现象：  
（1）对于需求描述较长或技术背景复杂的咨询，自动摘要有助于快速抓住关键信息，便于研发人员在阅读邮件时迅速了解用户核心诉求；  
（2）统一的邮件模板使得不同咨询内容在呈现形式上保持一致，便于后续归档与检索；  
（3）对于普通用户而言，能够在发送前预览由 AI 生成的摘要，有助于发现遗漏信息并进行补充，从而间接提升需求表达的完整性。  
综上，智能咨询模块在实验规模下验证了其对工程沟通效率与用户体验的积极作用。

6.6 本章小结

本章围绕智程导航、智眸千析、智航监控与智能咨询及自动邮件四个核心模块，从路径规划效果与性能、人物多特征识别精度与延迟、多模态问答质量、监控状态与告警链路表现以及咨询摘要可读性与解析鲁棒性等方面进行了实验评估与分析。实验结果表明：在给定的硬件与数据条件下，系统能够在保证基本实时性与可用性的前提下，提供较为合理的路径规划结果、较高精度的人物属性识别、可解释的监控大屏展示以及质量可控的咨询摘要输出。  
同时，实验也揭示了当前系统在复杂场景下仍存在的局限性，如语义解析偶发不稳定、极端光照条件下的识别误差、多模态链路中的误差传播以及监控模块在真实多源数据接入方面的约束等问题，为下一章的总结与未来工作展望提供了依据。
