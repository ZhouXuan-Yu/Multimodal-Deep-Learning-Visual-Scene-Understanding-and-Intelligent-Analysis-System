三、系统总体设计

3.1 系统设计目标与需求分析

3.1.1 系统设计目标

本系统面向无人机在安防与巡检等场景中的典型应用需求，综合考虑智能路径规划、人物多特征识别、无人机状态与视频监控以及智能咨询与自动邮件发送等功能，设计目标主要包括以下几个方面：

（1）**一体化功能集成**。在统一的平台中集成智程导航（路径规划）、智眸千析（人物识别）、智航监控（状态与视频监控）与智能咨询模块，避免传统系统中各功能孤立部署、数据割裂的问题，为用户提供连贯的任务规划、监控与信息分析体验。

（2）**自然语言驱动的人机交互**。通过引入本地部署的大规模语言模型与云端大模型服务，使用户能够通过自然语言描述任务需求，无需掌握复杂的专业参数配置；系统自动完成语义解析与参数抽取，实现“用语言驱动路径规划与信息咨询”的交互模式。

（3）**多源模型与服务协同**。将本地 LLM、深度视觉模型、高德地图 API 与 EmailJS 等第三方服务进行有机融合，构建“模型—服务—前端界面”协同工作流，在保证系统可扩展性的同时，兼顾开发效率与部署灵活性。

（4）**良好的可视化与可用性**。在前端采用富交互设计，对路径规划结果、人物识别结果与咨询摘要信息进行直观可视化展示，并通过地图、图像标注、表单与对话界面等多种形式提升用户操作体验。

3.1.2 功能需求分析

根据前期需求调研与项目实践，系统功能需求主要体现在以下几个方面：

（1）**智程导航功能需求**。系统需要支持用户通过自然语言描述或表单输入的方式提出路径规划需求，包括起点、终点、途经点、出行偏好（避拥堵、避收费、优先高速等）等信息；系统应能够调用本地大模型对自然语言需求进行解析，生成结构化规划参数，并结合高德地图 Driving API 生成多策略推荐路线；前端需支持路线列表展示、地图折线路径可视化以及路线详情查看等功能。

（2）**智眸千析功能需求**。系统需要支持用户上传图片（或视频帧），由后端视觉模块完成人物检测与多属性识别，包括年龄段、性别、上衣颜色、下衣颜色等；识别结果应以统一结构返回前端，并通过人物框选、高亮与标签展示等方式可视化呈现；在此基础上，系统还应预留与 LLM 联动的接口，用于对识别结果进行自然语言解释或支持按描述检索人物等高级功能。

（3）**智航监控功能需求**。系统需要提供无人机及地面监控设备的统一监控与指挥界面，包括设备在线状态、健康度（温度、电量、信号强度等）可视化、多路视频源的分屏展示与切换、巡逻任务的创建与执行情况展示等功能；同时，应支持对特定目标（如指定车牌号车辆）的持续监控与告警，在识别到目标事件时，能够通过可视化标记和邮件通知等方式提醒运维人员，从而实现对“航迹—设备—视频”多层信息的联动监控。

（4）**智能咨询与自动邮件功能需求**。系统需要提供联系与咨询页面，支持用户填写个人信息与自由文本需求描述；前端通过调用云端大模型服务，对用户需求进行摘要与结构化分析，生成适合邮件发送的咨询内容；随后利用 EmailJS 等第三方服务，将咨询摘要与原始信息封装为标准化邮件发送至预设邮箱，实现自动化信息传递。

（5）**系统管理与扩展需求**。系统应具有良好的模块化设计，便于后续替换或扩展算法模型（如更换路径规划模型或视觉模型），并支持增加新的业务模块（如无人机任务调度、告警联动等），同时在接口设计上保持统一与规范，降低维护与升级成本。

3.1.3 性能与非功能性需求

除核心业务功能外，系统在性能与非功能性方面也提出了相应要求：

（1）**响应时间与交互流畅性**。路径规划查询、人物识别推理与咨询摘要生成等关键操作应在可接受时间内完成，以保障用户交互流畅度。对于常见场景，应通过合理的模型部署与接口优化，将响应时间控制在秒级范围。

（2）**可靠性与鲁棒性**。在高德地图 API、云端大模型或邮件服务出现异常时，系统应具备基本的错误处理与回退机制，如提示用户稍后重试或采用默认策略规划路线，以保障核心功能的基础可用性。

（3）**安全性与隐私保护**。系统在处理用户上传的图像、表单信息以及任务数据时，应遵循最小化原则，避免无关数据存储与泄露；对涉及密钥与访问令牌的配置，应采用环境变量或后端代理方式进行保护，防止在版本库与前端代码中直接暴露。

（4）**可维护性与可扩展性**。系统架构应遵循分层与解耦原则，前端、后端、算法模型层与第三方服务之间通过明确接口协同工作，以便后续迭代开发与功能扩展。

3.2 系统总体架构设计

3.2.1 分层架构设计

结合项目的功能与技术栈，本文将系统总体架构划分为前端交互层、后端服务层、算法模型层、数据与第三方服务层以及系统集成层等五个主要层次，各层职责如下：

（1）**前端交互层**。主要由基于 Vue 3 的 Web 前端组成，包括智程导航页面、智眸千析页面以及联系与咨询页面等。该层负责用户界面展示、交互逻辑处理与部分轻量数据预处理，通过调用后端 API 或第三方服务实现业务功能。

（2）**后端服务层**。基于 FastAPI 等框架实现，主要承担路径规划服务、图像识别服务与统一聊天/任务路由服务等功能。该层对外暴露 RESTful 接口，负责接收前端请求、进行参数校验与业务逻辑处理，并与算法模型层、本地 LLM 与第三方服务进行交互。

（3）**算法模型层**。包括本地部署的大规模语言模型、本地视觉模型（如 YOLO、ResNet 等）以及与之相关的推理封装模块。该层对上提供统一的推理接口，使后端服务无需关注模型内部细节，只需根据业务需求构造输入并解析输出。

（4）**数据与第三方服务层**。主要包括本地数据库（如 SQLite）用于存储任务记录、日志与配置等信息，以及高德地图 API、DeepSeek Chat API、EmailJS 邮件服务等外部资源。该层通过标准化接口为上层服务提供地理信息、语言生成能力与邮件发送能力。

（5）**系统集成层**。负责前后端接口规范定义、各模块之间数据格式与调用协议的统一，以及跨模块业务流程的编排。通过该层，将智程导航、智眸千析、智航监控与智能咨询模块有机组合，形成完整的业务闭环。

3.2.2 系统总体架构描述

从整体上看，系统采用“前端富交互 + 轻量后端服务 + 本地模型 + 外部 API”的混合式架构。用户在浏览器端通过不同页面与功能模块进行交互：在智程导航界面中输入自然语言需求或选择起止点，在智眸千析界面上传图像并查看识别结果，在联系界面填写咨询表单。前端根据不同功能构造请求，通过统一的 HTTP 接口将数据发送至后端服务或第三方服务。

后端服务接收到请求后，根据内容类型与路由规则，将请求分发至对应的业务处理单元：对于路径规划请求，调用本地大模型进行语义解析，再调用高德地图 API 获取路线结果；对于图像识别请求，将上传图像交由视觉模型推理，并对结果进行结构化封装；对于统一聊天与任务请求，则根据 content.type 等字段决定调用哪一类子服务。最终处理结果以统一的数据结构返回前端，由前端完成地图渲染、结果展示与交互反馈。

3.2.3 模块划分与关系

在总体架构下，系统的核心业务模块可以划分为以下若干子系统：

（1）**智程导航子系统**。包括前端的路径规划页面与地图组件、后端的路线规划路由与聊天路由、本地 LLM 模块以及高德地图服务。其主要关系为：前端通过聊天或表单将用户需求发送至后端，后端调用本地 LLM 完成语义解析与参数抽取，再调用高德 API 进行多策略路径搜索，并将结果返回前端进行可视化展示。

（2）**智眸千析子系统**。包括前端的人物识别页面与图像展示组件、后端的图像识别路由与图像分析聊天接口、本地视觉模型以及可能的多模态 LLM。其主要关系为：前端上传图片至后端，后端调用检测与属性识别模型，输出统一的 `persons` 结构；在需要进行自然语言解释时，将识别结果与用户问题一并输入 LLM，生成可读性强的说明文本。

（3）**智航监控子系统**。主要由前端的数据监控大屏（包括设备状态面板、视频监控组件、无人机巡逻面板与地图追踪组件）、后端的车牌监控与告警路由、统一的设备与任务状态数据源等组成。该子系统一方面从后端服务与第三方识别服务中汇聚无人机与摄像设备的状态信息，对电量、信号、温度和健康度等指标进行聚合与分级展示；另一方面通过多路视频流与车牌监控接口，对交通路口、森林火灾、夜间街道等典型场景进行可视化监控，并在检测到目标车牌或异常事件时触发告警与邮件通知，从而为无人机任务提供“作业过程中的安全监控”能力。

（4）**智能咨询与自动邮件子系统**。主要由前端的联系页面、云端大模型服务与 EmailJS 邮件服务组成。前端负责收集用户信息与需求描述，构造 Prompt 调用云端 LLM 获取结构化咨询摘要，再将结果与用户信息一起通过 EmailJS 模板发送至指定邮箱，实现端到端的自动化咨询流程。

各子系统虽然面向不同的业务场景，但在系统架构层面共享部分基础设施与设计原则，例如统一的前端组件库、后端接口风格、模型调用封装以及错误处理机制，从而形成一个可复用、可扩展的综合应用平台。

3.3 主要功能模块设计

3.3.1 智程导航模块设计

（1）**功能结构设计**。智程导航模块主要包括需求输入与解析、路线规划与优化、结果展示与交互三个子模块。需求输入与解析子模块接收用户自然语言或结构化输入，调用本地大模型完成起终点、途经点与出行偏好的抽取；路线规划与优化子模块基于解析结果调用高德地图 API 获取多条候选路线，并根据时间、距离、费用等指标进行排序与标注；结果展示与交互子模块则负责在地图上绘制路径折线、显示路况信息，并提供路线切换与详情查看功能。

（2）**数据流设计**。用户在前端输入需求后，前端通过统一接口将请求发送至后端聊天路由或专用规划接口；后端首先将自然语言内容与预设 Prompt 组合，发送至本地 LLM，获得结构化路线参数；随后调用高德地图 Driving API 获取实际道路网络下的可行路径，并将路线信息整合为 `RouteResponse` 结构返回前端。前端解析该结构，更新推荐路线列表与地图渲染状态，实现从自然语言到路径可视化的完整数据流。

（3）**交互设计要点**。在交互层面，智程导航模块需要支持多条路线对比展示、路线细节查看以及必要的用户干预（如手动调整起终点或途经点）。同时，应提供适当的错误提示与状态反馈，例如在高德 API 调用失败或大模型解析异常时，向用户提示重试或采用默认策略。

3.3.2 智眸千析模块设计

（1）**功能结构设计**。智眸千析模块主要由图像上传与预处理、人物检测与属性识别、结果展示与多模态交互三个子模块构成。图像上传与预处理子模块负责接收用户上传的图像文件，并完成尺寸调整与格式校验；人物检测与属性识别子模块基于 YOLO、ResNet 等模型完成人物位置检测与年龄、性别、服装颜色等属性的联合识别；结果展示与多模态交互子模块则在前端以框选与标签形式展示识别结果，并在需要时将识别结果与用户问题提交给 LLM，生成自然语言解释或检索结果。

（2）**数据流设计**。用户在前端选择或拖拽上传图片后，前端构造包含图像文件的请求并发送至后端图像识别接口；后端读取图像，调用相应的检测与识别模型，并将每个人物的边界框坐标、年龄段、性别、上衣颜色与下衣颜色等信息封装为统一 JSON 结构返回前端。若用户进一步提出自然语言问题，前端则将该问题与识别结果一并发送至图像分析聊天接口，由 LLM 生成解释性回答后返回前端显示。

（3）**交互设计要点**。在交互层面，智眸千析模块应支持对单个人物或全部人物的属性查看，并通过颜色标注或图例提示不同属性；对于识别不确定或置信度较低的结果，可以在界面上进行视觉弱化或标注，以提醒用户谨慎使用；同时，为避免频繁上传大体积图片造成性能问题，可在前端进行简单的压缩或尺寸限制。

3.3.3 智能咨询与自动邮件模块设计

（1）**功能结构设计**。该模块主要由用户信息采集、咨询内容智能生成与邮件发送三个子模块组成。用户信息采集子模块通过表单收集用户姓名、联系方式、感兴趣的功能类型与自由文本需求描述；咨询内容智能生成子模块构造含有项目功能说明与用户需求的 Prompt，调用云端大模型生成包含标题、大纲、建议与解决方案等内容的结构化摘要；邮件发送子模块则将摘要结果与用户信息映射到预设的邮件模板，并通过 EmailJS 完成邮件发送。

（2）**数据流设计**。用户在前端填写并提交表单后，前端首先调用云端大模型接口获取咨询摘要与结构化内容；若模型调用成功，前端将摘要与用户基础信息一并封装为模板参数，调用 EmailJS 的发送接口完成邮件投递；若模型调用失败或返回格式不符合预期，则前端使用预设的回退模板生成基础邮件内容，确保咨询信息仍然能够发送到指定邮箱。

（3）**交互设计要点**。在交互设计上，联系页面应对输入字段进行基本校验，并在发送过程中提供加载状态与成功/失败反馈；对于发送失败的情况，应提示用户检查网络或稍后重试，必要时提供备用联系方式。同时，可以在界面上适当展示由大模型生成的摘要内容，让用户在发送前进行确认，从而提高邮件内容的准确性与可读性。

3.3.4 智航监控模块设计

（1）**功能结构设计**。智航监控模块主要由设备与无人机状态面板、视频监控与告警展示、巡逻任务编排与执行监控三个子模块构成。状态面板聚合来自无人机与多类固定监控设备的健康度信息，对在线/离线状态、温度、电量、信号强度等关键指标进行分类统计与可视化展示；视频监控子模块以多路分屏形式呈现不同场景下的监控画面，并在前端叠加车牌识别、人物检测、火灾监测等高层语义结果与告警标记；巡逻任务子模块则提供无人机巡逻任务的配置、AI 推荐与路径区域选择界面，模拟多机协同巡逻过程中的状态变化。

（2）**数据流设计**。在数据流层面，智航监控模块一方面从后端接口获取无人机与监控设备的状态数据（包括预定义模拟数据与实际监控数据），通过统一的数据结构在前端进行定时刷新与动画展示；另一方面，通过车牌监控路由接收用户上传的图片或视频，对其进行车牌检测、车牌与车辆颜色识别以及目标车牌匹配，将识别结果以 JSON 形式返回前端，并在监控界面中以高亮框与文本标签的方式叠加显示。此外，部分监控任务还会通过邮件通知模块对严重告警进行二次推送，实现“监控—识别—通知”的闭环。

（3）**交互设计要点**。在交互层面，智航监控模块强调“一屏多维”的综合展示与可操作性：用户可以在状态面板中按设备类型或状态过滤设备列表，在视频监控区域切换视图布局和监控类型，在巡逻任务面板中配置任务目标、选择无人机并绘制巡逻区域。对于识别出的目标车牌或高风险事件，界面通过颜色编码、闪烁动画与提示文本进行凸显，并提供检索与回放入口，以便运维人员快速定位问题源头，实现对无人机任务执行过程的直观监管与辅助决策。

3.4 关键业务流程设计

3.4.1 智程导航业务流程

智程导航的典型业务流程可概括为以下几个步骤：  
（1）用户通过前端界面输入自然语言出行需求或选择起终点与途经点；  
（2）前端将需求封装为请求，发送至后端聊天路由或规划接口；  
（3）后端调用本地大模型，对自然语言描述进行语义解析与参数抽取，得到结构化路线参数；  
（4）后端基于解析结果调用高德地图 Driving API，获取多条候选路线及其详细信息；  
（5）后端将路线数据整合为统一响应格式返回前端；  
（6）前端对路线进行列表与地图可视化展示，并支持用户选择、对比与调整。

3.4.2 智眸千析业务流程

智眸千析的典型业务流程包括：  
（1）用户在前端选择或拖拽上传监控图像或视频帧；  
（2）前端对图像进行基本校验与预处理后，构造请求发送至后端图像识别接口；  
（3）后端加载或调用已加载的视觉模型，对图像中的人物进行检测，并对每个人物的年龄、性别与服装颜色等属性进行推理；  
（4）后端将识别结果封装为包含人物编号、边界框、属性与置信度等字段的 JSON 结构返回前端；  
（5）前端在图像上绘制人物框，并以标签或侧边栏形式展示属性信息；  
（6）若用户进一步提出自然语言问题，前端将识别结果与问题发送至图像分析聊天接口，由 LLM 生成解释或检索结果后展示给用户。

3.4.3 智能咨询与自动邮件业务流程

智能咨询与自动邮件的业务流程主要包括：  
（1）用户在联系页面填写个人信息、选择关注功能类型并输入需求描述；  
（2）前端将上述信息与预设项目功能说明一起构造 Prompt，调用云端大模型接口获取结构化咨询摘要；  
（3）前端对大模型返回内容进行解析，将标题、大纲、建议等字段与用户信息组合，填充至预设邮件模板；  
（4）前端通过 EmailJS 或类似服务提交发送请求，由第三方服务完成邮件投递；  
（5）系统在界面上反馈发送结果，并可在本地记录必要的发送状态信息，方便后续查询与统计。

3.5 本章小结

本章在前两章研究背景与相关技术分析的基础上，从系统设计目标与需求出发，给出了无人机综合应用平台的总体架构与主要功能模块划分。通过分层架构设计与子系统划分，明确了智程导航、智眸千析、智航监控以及智能咨询与自动邮件模块在系统中的角色与交互关系，并对各模块的数据流与关键业务流程进行了说明。为后续第四章中对核心算法设计与实现细节的展开奠定了整体框架基础。
