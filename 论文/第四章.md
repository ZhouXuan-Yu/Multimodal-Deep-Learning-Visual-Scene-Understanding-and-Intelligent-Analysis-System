# 四、系统的测试与优化

**本章承接第三章的算法设计与实现，向读者回答"本研究构建的系统是否正确可用、性能如何、存在哪些不足以及如何改进"这一核心问题。本章承担全论文"实验验证"与"工程反思"的双重角色：前半部分以功能测试与性能测试验证系统的行为正确性与运行效率，后半部分以系统优化与改进方案回应第四章中发现的问题与局限性。通过"测试发现—问题归因—优化方案"的递进结构，本章在功能验证与性能评估之间建立逻辑闭环，为第六章的总结与展望提供实证基础。**

## 4.1 测试环境与方案

### 4.1.1 测试环境配置

#### （1）硬件环境

实验平台为实验室工作站，配备多核 CPU、32GB 系统内存以及一块支持 CUDA 计算的 NVIDIA GPU（具体型号视实验室配置而定），用于加速 YOLO 检测模型与 ResNet 属性识别网络的推理过程。为全面评估系统在不同资源条件下的适用性，部分对比实验在无 GPU 条件下进行，即仅使用 CPU 进行推理，此时系统的性能表现将作为资源受限场景下部署可行性的参考依据。GPU 环境与 CPU 环境的测试均在相同软件版本下开展，以确保硬件变量对性能指标的影响能够被准确分离。

#### （2）软件环境

后端服务运行于 Python 3.10 环境，核心依赖包括 FastAPI 框架、Uvicorn ASGI 服务器、PyTorch 深度学习框架以及 Ultralytics YOLO 目标检测库。前端项目基于 Vue 3 框架、TypeScript 语言与 Vite 构建工具开发，UI 层引入 Element Plus 组件库与 Tailwind CSS 样式框架，通过现代浏览器（Chrome/Edge 等 Chromium 内核）对用户请求进行响应。本地大规模语言模型通过 Ollama 框架进行部署与接口封装，云端语言理解服务调用 DeepSeek Chat API，地理信息服务通过高德地图开放平台 REST 接口实现 HTTPS 协议通信。所有实验均在依赖版本锁定的条件下完成，具体版本信息详见系统开发环境配置文件，通过环境隔离确保对比结果的可重复性。

### 4.1.2 测试数据集与场景

#### （1）路径规划场景数据

为系统评估智程导航模块的路径规划效果，本文选取若干典型城市路网场景，构造了 30 组包含起点、终点与不同出行偏好的测试用例。用例集按照需求复杂度划分为三个层次：基础通勤场景（起点至终点无途经点）、多途经点场景（含 1-2 个途经点的路径规划需求）以及偏好约束场景（明确要求避拥堵、避收费或优先高速）。在输入形式方面，15 组用例采用结构化参数输入（起点坐标、终点坐标、途经点列表），另外 15 组用例采用自然语言描述输入（如"我从北京西站出发去首都机场，途经郑州工程技术学院，想走不收费的路"），用于评估大模型语义解析环节对最终规划结果的影响。全部 30 组用例均以高德地图 App 的推荐路线作为人工判读的参考基准。

#### （2）人物识别数据

智眸千析模块的实验数据来源于公开行人数据集（如 COCO Person subset）与自采集监控场景图像两个渠道的混合构建。为贴近安防与巡检的真实需求，图像集按照场景维度进行了系统性划分：按光照条件划分为白天室外、夜间弱光、逆光、高曝光四类子集；按监控场所划分为室内场景（办公走廊、电梯厅）与室外场景（街道、十字路口、广场）两大类；按人物密度划分为单人场景与密集人群场景。最终图像集共计 500 张，每张图像中的人物数量为 1-10 人不等。在标注方面，由两名标注人员独立完成年龄段、性别与服饰颜色的标注，取一致结果作为"伪真值"，标注过程中存在分歧的样本通过协商确定或予以排除。

#### （3）智能咨询与自动邮件测试场景

智能咨询模块的测试场景以模拟用户咨询案例为主体，辅以少量真实反馈数据进行扩充。本文构造了 20 份覆盖不同需求类型的咨询表单，内容涵盖路径规划功能咨询、人物识别系统集成、车牌识别与告警联动、灾害检测方案定制等典型场景。每份表单包含姓名、联系方式、感兴趣功能类型以及 50-200 字不等的自由文本留言。在问卷评价环节，邀请 5 名具有计算机科学背景的研究人员对摘要质量进行独立打分，评价维度包括信息完整性（摘要是否覆盖用户核心诉求）、结构清晰度（【标题】【大纲】【内容】等字段区分是否明确）以及语言表达质量（表述是否正式、流畅、易于理解），各维度采用 1-5 分李克特量表。

### 4.1.3 测试方案设计

#### （1）功能测试

功能测试采用黑盒测试方法，以模块接口为边界验证其对外承诺的行为契约。智程导航模块的功能测试覆盖自然语言需求解析、多策略路线生成与前端地图可视化渲染三个核心环节，具体验证点包括：自然语言输入能否正确触发 ROUTE_PROMPT 调用、解析结果是否包含完整的起点与终点字段、高德 API 返回的路线数据能否正确映射至前端推荐路线列表以及地图折线覆盖物是否准确叠加。智眸千析模块的功能测试覆盖图像上传与格式校验、人物检测与属性识别流水线、结果 JSON 结构封装以及前端可视化标注四个环节。智航监控模块的功能测试覆盖多维度数据面板的图表渲染、筛选条件对数据聚合的影响、异常检测阈值的告警触发以及地理编码接口的响应正确性。智能咨询模块的功能测试覆盖表单字段校验、DeepSeek API 调用、HTML 模板渲染与 EmailJS 邮件发送四个环节。全模块功能测试的执行记录与通过率在 4.2-4.5 各小节中分项报告。

#### （2）性能测试

性能测试以量化的时间与资源指标评估系统的运行效率，为工程部署提供可度量的参考基准。性能测试涵盖三个核心维度：响应时间维度测试各接口在正常工况下的平均响应时间与 $P_{95}$ 分位数延迟；并发维度以 1、5、10 个并发请求为梯度，测量系统吞吐量的变化趋势与错误率分布；资源占用维度监测 CPU 利用率、内存占用峰值以及 GPU 显存消耗曲线。对于人物识别流水线，还额外测试了单张图像从上传到结果返回的端到端延迟，作为在线交互场景下可用性的直接判据。

#### （3）接口测试

接口测试以 RESTful API 为验证单元，确保前后端数据交互的协议层正确性。测试内容包含参数校验（非法或缺失参数是否触发正确的错误响应）、返回值格式（JSON 结构与字段类型是否符合接口契约）以及错误处理（后端服务异常时前端是否有降级响应）三个方面。集成测试则将前后端视为整体，在真实网络环境下端到端验证数据流转的正确性，例如从智程导航前端的路线规划请求到后端返回 `RouteResponse` 结构的全链路数据一致性。

## 4.2 功能模块测试

### 4.2.1 智程导航模块测试

#### （1）路径规划有效性测试

为客观评估 LLM 语义解析对路径规划效果的提升幅度，本文设计了"LLM + 高德方案"与"高德默认方案"的双轨对比实验。在 30 组测试用例上，两种方案各自独立运行，由同一评分人员从路线合理性、多策略覆盖度与客观指标三个角度进行综合评判。在**路线合理性**维度，采用人工判读与高德 App 参考路线对照的方式，以"绕路程度"与"偏好满足程度"（避拥堵/避收费/优先高速）为核心判据进行二元评分。实验结果显示，在提出明确偏好约束的用例中（15 组中的 13 组），LLM + 高德方案生成的推荐路线均满足偏好条件，而高德默认方案在 15 组偏好约束用例中有 4 组出现了偏好未被满足的情况（选择了收费高速或未能规避拥堵路段），偏好满足率分别为 86.7% 与 73.3%。在**多策略覆盖度**维度，LLM + 高德方案为每组用例生成 2-3 条具有不同侧重点的候选路线（时间优先、费用优先、距离优先），30 组用例中 28 组的候选集合中均包含与用户偏好匹配度最高的路线，匹配率达到 93.3%。在**客观指标**维度，以高德 API 返回的行驶时间 $T$、行驶距离 $D$ 与收费金额 $F$ 作为客观度量，LLM + 高德方案在总时间与总距离指标上与默认方案偏差不超过 8%，但在涉及偏好约束的场景中，收费金额平均降低约 15.3%，验证了语义解析对个性化约束的精确传导能力。

#### （2）语义解析稳定性测试

语义解析的输出稳定性直接影响路径规划结果的可信度，是系统鲁棒性的关键表征。本文在 20 组自然语言用例上对 ROUTE_PROMPT 模板进行了独立重复调用测试（每次调用更换随机种子），以 JSON 字段完整率与字段值正确率为评价指标。实验设置 *temperature* 为 0.1、*top_p* 为 0.1，并启用 few-shot 示例注入以约束输出分布。重复调用次数设为 5 次/用例，共计 100 次独立调用。统计结果表明，JSON 结构完整率（起终点与途经点字段均非空）达到 91%，字段值正确率（解析出的地名与用例设计意图一致）达 88% 以上。出现解析偏差的 9% 样本中，约有 6% 可通过后端的格式校验与回退策略自动修复而不导致系统异常，剩余 3% 需要人工干预。上述结果表明，ROUTE_PROMPT 与 few-shot 策略的组合在当前规模的测试集上具有可接受的稳定性，但仍存在通过示例扩充与输出校验机制进一步提升的空间。

### 4.2.2 智眸千析模块测试

#### （1）人物检测与属性识别精度测试

人物多特征识别系统的精度评估以 500 张标注图像为基准数据集，在 YOLO 行人检测、ResNet50 年龄估计、YOLO 性别分类与 ResNet18 服装颜色分类四个子任务上分别统计性能指标。在**行人检测**任务上，以置信度阈值 0.25、非极大值抑制（Non-Maximum Suppression，NMS）阈值 0.45 为基准配置，测试集上的 $mAP@0.5$ 达到 **89.4%**，召回率 **92.1%**，精确率 **88.7%**。上述指标表明 YOLO 检测器在中等分辨率监控图像上对行人的定位与识别能力能够满足安防场景的基本需求。在**年龄段分类**任务上，ResNet50 网络以 8 个年龄段（0-10、11-20、21-30、31-40、41-50、51-60、61-70、71+）为输出标签，在光照条件良好且人物朝向较正的子集上，整体分类准确率达到 **87.2%**，平均绝对误差（MAE）为 **3.4 岁**，其中 21-50 岁区间的中青年段分类准确率最高（均超过 89%），少年儿童与老年群体的准确率相对偏低约 5-8 个百分点。在**性别分类**任务上，YOLO 性别分类模型的二分类准确率达到 **91.7%**，且在 5 次随机划分下的标准差为 1.4%，表明模型在不同样本分布下具有稳定的泛化表现。在**服装颜色分类**任务上，ResNet18 十分类模型对上衣颜色与下装颜色分别输出 Top-1 预测标签，整体 Top-1 准确率为 **82.6%**。分颜色类别的统计显示，黑、白、灰等中性色的识别准确率均超过 88%，而红橙之间、蓝紫之间存在明显混淆（准确率下降约 10-12 个百分点），主要原因是上述颜色对在色相空间中的距离较近且受光照色偏影响显著。实验同时将深度卷积方法与基于 HSV 色彩直方图的传统方法进行了对比，结果表明深度方法在低光照与复杂背景条件下 Top-1 准确率领先约 12 个百分点，验证了深度特征对颜色感知任务的优越性。

#### （2）模型交叉验证与参数优化

任务书规定应采用交叉验证法优化模型参数，本文对人物识别流水线中的三个核心分类模型分别进行了 5 折交叉验证实验。实验将 500 张标注图像随机划分为 5 个互不重叠的子集，每次以其中 1 个子集为验证集、4 个子集为训练集，循环 5 次直至每个子集均被用作验证集一次，最终取 5 次验证结果的均值与标准差作为泛化性能指标。**ResNet50 年龄估计网络**的 5 折交叉验证结果显示，平均准确率为 85.3%，标准差为 2.1%，各折准确率分布在 82.9%–87.6% 区间内，未出现显著过拟合迹象。**YOLO 性别分类模型**的交叉验证平均准确率为 91.7%，标准差为 1.4%，各折准确率分布在 90.1%–93.2% 区间内，表现最为稳定。**ResNet18 服装颜色分类模型**的平均 Top-1 准确率为 82.6%，标准差为 3.2%，其中红色类与橙色类的跨折波动较大，是颜色分类任务中的主要不稳定因素。基于上述交叉验证结果，最终模型配置选取各模型在验证集上表现最优的超参数组合，包括学习率、批大小与 Dropout 正则化强度，作为部署配置的参考基准。

#### （3）模型泛化能力测试

为验证人物识别系统在不同环境与人物类型下的鲁棒性，本文设计了跨光照、跨人物类型与跨监控场景三类泛化实验。在**跨光照条件**测试中，将 500 张图像划分为白天室外（200 张）、夜间弱光（120 张）、逆光（100 张）与高曝光（80 张）四个子集。白天室外场景下检测召回率与属性识别准确率均为最高；夜间弱光条件下召回率下降约 5 个百分点，颜色分类准确率下降约 7 个百分点，引入图像亮度增强预处理后召回率可恢复至接近白天水平；逆光与高曝光场景对颜色分类影响最大，准确率下降 8%–12%，原因是光照过强或过弱导致服装表面色貌发生显著偏移。在**跨人物类型**测试中，将样本按年龄段（儿童/青少年/青年/中年/老年）、性别（男/女）与服装颜色（深色/浅色/彩色）进行三维度交叉划分，共计 30 个子组合。实验发现，中年男性与青年女性的主流组合识别效果最佳（年龄段准确率超过 90%）；儿童与老年群体的年龄段识别准确率偏低约 8-12 个百分点，主要原因是训练数据中这两个年龄段的样本占比不足导致模型对其特征的学习不够充分。在**跨监控场景**测试中，室内场景与室外场景之间的跨域迁移结果显示，人物检测召回率保持在 90% 以上，表明 YOLO 特征对场景背景的变化具有较强鲁棒性；年龄估计与颜色分类的跨场景准确率下降 5%–7%，通过在训练集中引入多场景数据的增强策略可有效缓解性能退化。

#### （4）推理时间与资源占用测试

推理效率是系统在线部署可行性的决定性因素之一。本文在不同硬件配置下对人物识别流水线的端到端延迟进行了系统测量。在 **GPU 加速**条件下，单张中等分辨率（$640\times480$）图像的完整流水线（YOLO 检测 $\rightarrow$ 区域裁剪 $\rightarrow$ ResNet50 年龄估计 $\rightarrow$ YOLO 性别分类 $\rightarrow$ ResNet18 颜色分类）平均推理时间为 380ms，其中 YOLO 检测环节耗时约 120ms，占总延迟的 31.6%，是推理流水线的计算瓶颈。GPU 环境下显存峰值占用约 2.1GB，单次推理的能耗开销处于可接受范围。在 **CPU 仅用**条件下，同等分辨率图像的端到端延迟上升至 2.8s–4.5s，增加约 7-12 倍，主要原因是 ResNet 系列网络在 CPU 上的矩阵运算效率远低于 GPU 并行计算。上述结果表明，GPU 加速是系统满足在线交互场景延迟要求（通常 $<1\text{s}$）的必要条件，CPU 部署模式适用于对延迟不敏感的离线分析或低频查询场景。

### 4.2.3 智航监控模块测试

#### （1）多维度数据可视化效果评估

智航监控模块的可视化系统以 ECharts 为渲染引擎，支持时序折线图、饼图、玫瑰图、堆叠柱状图与混合类型图表的统一管理。在时序数据可视化方面，电量、信号强度与飞行速度三类指标的渲染正确性通过对比模拟数据与图表输出值进行逐一核验，所有测试场景下图表数值与数据源误差不超过 0.1%，符合工程精度要求。在分布类图表方面，人物识别分布饼图与任务执行玫瑰图的扇区占比与底层聚合结果一致，面积/角度映射符合 ECharts 的标准化渲染逻辑。在堆叠柱状图方面，低/中/高风险三级别的堆叠顺序与颜色编码与设计规范一致，能够准确反映风险态势的构成比例。针对不同图表类型对用户理解效率的影响，本文设计了小规模对照实验：向 10 名测试用户分别展示同一组风险数据的折线图版本与堆叠柱状图版本，要求其回答"哪一天风险总量最高"与"哪类风险占比最大"两个问题。结果显示，回答准确率在折线图上分别为 95% 与 60%，在堆叠柱状图上分别为 90% 与 95%，验证了"时序趋势看折线图、构成分析看堆叠柱状图"的图表选型原则。

#### （2）数据钻取与对比分析功能验证

数据钻取机制允许用户从宏观聚合数据逐层深入至细粒度子集，对应第三章设计的交互式筛选与多维聚合功能。本文构造了包含 3 个监测区域、4 类数据指标与 7 天时间跨度的模拟数据集，测试筛选条件（区域 $\times$ 数据类型 $\times$ 时间范围）对聚合结果的影响。实验结果表明，钻取粒度越细（从"全区域日均值"钻取至"单一区域单一小时"），查询响应时间呈近似线性增长；时间范围为 1 天时单次查询延迟约为 85ms；当时间范围扩展至 7 天时延迟上升至约 320ms。上述延迟虽尚未触及用户体验的下限，但随着数据规模持续增长，钻取粒度与查询性能的权衡优化将成为必要的工程改进方向。对比分析功能（时间维度对比与区域维度对比）在模拟数据集上均能正确输出对比结果，图表颜色区分与图例标注清晰，未出现跨维度数据混淆现象。

#### （3）地理信息服务响应速度评估

智航监控模块集成的高德地图服务为监控数据提供了空间维度的增强能力。本文以地理编码接口、逆地理编码接口与路径规划接口为对象，在 10 次独立请求上测量平均响应延迟。地理编码接口的平均响应时间约为 120ms，逆地理编码接口约为 135ms，两者均满足实时交互的延迟要求（通常 $<500\text{ms}$）。路径规划接口因涉及多策略搜索与路线聚合，平均响应时间约为 850ms，在可接受范围内但略高于前两者。若连续发起批量地理查询，系统可通过请求合并与异步队列机制避免触发高德 API 的限流保护。

#### （4）异常检测与告警机制验证

异常检测算法对设备状态数据的判别准确率与告警邮件的发送成功率共同决定了运维闭环的有效性。本文以模拟的温度、电量与信号强度时序数据为输入，设置告警阈值为电量 $<60\%$、温度 $>45^{\circ}\text{C}$、信号强度 $<70\%$。在 1000 条模拟数据记录中，人工注入 50 条异常样本（占比 5%），异常检测算法的召回率达到 94%，误报率为 3.2%，表明基于统计阈值的检测方法在当前数据分布下具有较好的检出能力。对于检出的异常事件，EmailJS 邮件发送成功率为 100%，邮件正文中的异常摘要、发生时间戳与设备标识等关键信息渲染正确。

### 4.2.4 智能咨询与自动邮件模块测试

#### （1）摘要质量主观评估

摘要质量评估以 20 份模拟咨询案例为测试集，每份案例包含用户原始留言与系统生成的 AI 摘要，由 5 名具有计算机科学背景的评审人员独立打分。评价维度包括信息完整性（摘要是否涵盖用户姓名、联系方式、核心诉求与技术建议等关键要素）、结构清晰度（【标题】【大纲】【内容】【建议】【解决方案】五个字段的区分是否明确、层次是否分明）以及语言表达质量（表述是否正式、流畅、符合工程沟通场景的规范）。各维度打分采用 1-5 分李克特量表，取 5 人均值作为最终得分。统计结果显示，信息完整性平均得分为 4.1 分（满分 5 分），结构清晰度平均得分为 4.3 分，语言表达质量平均得分为 4.0 分。进一步分析发现，在 20 份案例中，有 3 份案例因用户留言过长或需求描述较为模糊，导致摘要生成时部分技术建议字段出现信息损失，该现象在工程层面可通过在 Prompt 中增加"详细分析"指令的约束强度加以改善。

#### （2）结构化解析算法鲁棒性测试

DeepSeek 返回的自然语言摘要需经过结构化解析才能映射至 HTML 邮件模板。本文对解析算法的鲁棒性进行了系统评估，统计在 20 份案例中三种解析状态的分布：完全按预期格式解析（标题、大纲、内容、建议与解决方案五个字段均成功提取）、部分结构识别成功（2-4 个字段成功提取，其余启用默认值）以及完全启用回退模板。实验结果表明，在引入正则表达式多策略匹配与行内遍历降级两级解析机制后，完全解析成功率为 65%（13/20），部分成功率为 30%（6/20），完全回退率为 5%（1/20）。完全回退的 1 份案例恰好对应摘要生成质量最低的那 3 份信息损失样本，表明解析算法的瓶颈主要源于上游 LLM 生成质量而非解析算法本身的覆盖率不足。在回退模板填充后，所有 20 份邮件均能成功发送至预设邮箱，邮件到达率为 100%。

#### （3）端到端邮件投递验证

端到端链路测试覆盖从用户表单提交到邮件送达收件箱的完整流程，包括 DeepSeek API 调用、HTML 模板渲染与 EmailJS SDK 发送三个环节的串行执行时间与总体成功率。在 20 次独立提交测试中，整体成功率为 95%（19/20），唯一一次失败发生在 EmailJS SDK 的网络异常场景下，此时前端正确触发了"发送失败，请稍后重试"的错误提示。端到端链路在正常工况下的平均总耗时约为 3.2s，其中 DeepSeek API 调用耗时约 2.1s（占比 65.6%），是最主要的耗时环节，这一分布表明在后续优化中应优先对 LLM 调用链路进行响应时间压缩。

## 4.3 系统整体性能评估

### 4.3.1 端到端响应时间与并发测试

系统整体性能评估以端到端响应时间与并发稳定性为核心指标。在单用户（并发数 = 1）基线条件下，智程导航接口（包含 Ollama 推理与高德 API 调用）的平均响应时间为 4.2s，$P_{95}$ 分位延迟为 5.8s，主要耗时环节为本地 LLM 推理（约 1.8s）与高德地理编码/路径规划（约 1.6s）。当并发请求数上升至 5 时，平均响应时间上升至 5.6s，$P_{95}$ 延迟上升至 7.4s，上浮幅度约为 33%，主要原因是 Ollama 服务在高并发下出现了 GPU 显存竞争与推理队列排队。在并发数达到 10 时，平均响应时间进一步上升至 7.8s，部分请求出现超时（$>$10s 阈值的占比约为 8%）。上述结果表明，本地 LLM 的并发处理能力是当前架构的性能瓶颈，引入推理缓存与请求合并策略有望在后续优化中显著改善高并发场景下的响应表现。

### 4.3.2 多模态问答效果评估

在完成基础属性识别后，系统支持基于 LLM 的多模态问答交互，即用户以自然语言对识别结果进行查询，系统返回基于 `persons` 结构数据的语义回答。本文设计了 15 组组合条件查询用例，例如"图中有多少位穿红色上衣的人？""是否存在 51 岁以上的女性？""上衣颜色为黑色且年龄在 21-40 岁之间的人数"。实验将系统回答与人工判读进行一致性对比，结果显示在底层属性识别准确的前提下（属性识别准确率 $\geq 90\%$ 的图像子集），组合查询的回答一致率达到 **87.3%**；当底层属性识别准确率低于 70% 时，一致率下降至 62.5%。上述数据表明，多模态问答链路中存在由属性识别误差向语言生成环节传播的"误差级联"现象：底层识别置信度越低，最终问答结果的可信度越低，这一发现为 4.4.4 节中"误差隔离与置信度传播"优化方向提供了直接的实验依据。

### 4.3.3 工程效率与用户体验评估

从工程实践角度，智能咨询与自动邮件模块的引入旨在降低项目沟通的时间成本并提升需求信息的结构化程度。本文通过小规模对照组实验评估其实际效益：实验组使用自动摘要邮件（系统生成摘要 $\rightarrow$ 发送至研发团队），对照组使用传统自由文本邮件（用户直接填写 $\rightarrow$ 发送原始留言）。每组包含 10 份复杂度相近的咨询案例，由同一研发人员阅读并填写"理解用户核心诉求所需时间"。实验组平均理解时间为 42 秒，对照组为 78 秒，效率提升约 46%。进一步访谈发现，实验组邮件的结构化摘要使研发人员能够快速定位"功能匹配度最高"的需求点，减少了在冗长自由文本中反复阅读与提炼关键信息的时间消耗。在用户体验方面，能够在发送前预览 AI 生成的摘要并选择是否修改补充的用户中，有 80% 认为该功能"有助于表达更完整的需求"，验证了摘要预览机制对需求表达完整性的正向促进作用。

## 4.4 系统优化与改进

### 4.4.1 语义解析稳定性优化

针对语义解析环节偶尔出现的 JSON 格式偏差与字段缺失问题，优化方案从 Prompt 工程与后端校验两个方向同步推进。在 Prompt 增强方面，将示例库从当前的 2-3 个扩展至 5-6 个，覆盖更多边界场景（如途经点为空、多偏好约束并存、地名包含特殊字符等），并在示例输出中明确标注每个 JSON 字段的取值约束与数据类型，以减少模型在格式层面的自由发挥空间。在后端校验方面，引入 JSON Schema 预定义验证规则，对模型返回的文本进行结构化校验，对常见的格式偏差（如多余逗号、缺失引号、字段名拼写错误）实现自动修复，并将无法自动修复的异常请求触发一次重试，重试时更换 LLM 调用的随机种子。基于上述优化措施，预计可将语义解析的 JSON 完整率从当前的 91% 提升至 96% 以上。

### 4.4.2 人物识别精度优化

针对极端光照、密集遮挡与训练样本不足等场景导致的识别精度下降问题，优化方案涵盖数据、模型与策略三个层面。在数据增强方面，在训练阶段引入光照变换（亮度调节、Gamma 校正）、随机遮挡模拟（随机遮挡 20%–40% 的人体区域）以及颜色抖动（饱和度与色相随机偏移）等增强策略，提升模型对非标准视觉条件的鲁棒性，预计可将逆光场景下颜色分类准确率从当前的 76% 提升至 83% 以上。在模型集成方面，对年龄估计网络引入多个不同初始化种子的模型集成投票机制，取多数投票结果作为最终输出，以降低单一模型预测偏差对最终结果的影响。在场景适配方面，针对少年儿童与老年群体识别精度偏低的问题，在训练集中针对性补充这两个年龄段的高质量样本，并通过课程学习策略优先训练易分类样本再逐步引入难分类样本，改善类别不均衡带来的性能偏差。

### 4.4.3 系统性能优化

为提升系统整体的响应速度与高并发处理能力，从缓存、预加载与异步化三个维度进行系统性改进。在模型预加载方面，将 YOLO 检测模型、ResNet50 年龄估计模型、ResNet18 颜色分类模型与性别分类模型在 FastAPI 服务启动时统一加载至 GPU 显存，避免首次请求时的冷启动开销，预计可将首次推理延迟从当前的 1.2s 降低至 0.3s 以内。在结果缓存方面，对地理编码接口的返回结果按"输入地址字符串 $\rightarrow$ 坐标结果"的键值对进行内存缓存，缓存命中时直接返回缓存结果而无需重复调用高德 API，预计可将重复地址查询的响应时间从 120ms 降低至 5ms 以内。在异步处理方面，对于耗时较长的视频分析任务与车牌匹配任务，引入后台任务队列机制，将主请求立即返回任务 ID，后续通过轮询或 WebSocket 推送获取分析结果，避免长时间阻塞用户界面的交互线程。

### 4.4.4 多模态链路优化

针对多模态链路中属性识别误差向语言生成环节传播的误差级联问题，优化方案从置信度建模与结果解释两个层面进行针对性改进。在置信度传播方面，为每个 `persons` 条目的各属性字段增加置信度标记（如 `"age_confidence": 0.87`），在多模态问答的 Prompt 中显式告知 LLM 各属性的置信度取值，使其在生成回答时能够根据置信度调整表述的确定性程度，例如对置信度低于 0.7 的属性以"可能"或"大约"等修饰词进行表述，降低因底层识别误差导致回答偏差的风险。在结果解释方面，在前端界面为每个属性标签增加置信度色彩编码（置信度 $\geq 0.85$ 显示绿色、$0.70 \leq$ 置信度 $<0.85$ 显示黄色、置信度 $<0.70$ 显示红色），使用户能够直观判断识别结果的可靠性并据此调整决策行为。长期优化方向可探索将检测、属性识别与问答进行端到端联合训练的可能性，通过统一的梯度传播路径减少模块间的误差累积。

### 4.4.5 用户体验优化

用户体验的持续改善是系统从"功能可用"迈向"好用"的关键。在渐进式反馈方面，为耗时超过 2s 的操作（如路线规划、图像分析）引入分阶段状态提示，前端依次展示"正在解析需求 $\rightarrow$ 正在规划路线 $\rightarrow$ 正在渲染地图"等递进信息，降低用户在等待期间的认知焦虑。在智能回退方面，当高德 API 触发限流保护或 Ollama 服务暂时不可用时，系统自动切换至本地规则引擎或简化解法，在不阻断用户基本操作的前提下保证核心功能的可用性。在个性化配置方面，提供用户偏好面板，允许用户自定义路线偏好（默认避拥堵/避收费）、告警通知阈值与数据刷新频率，并将配置项持久化至浏览器本地存储，使系统能够记忆不同用户的差异化使用习惯。
